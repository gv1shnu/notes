Access control and IDOR

Difficulty: Medium

The most reliably-present bug in real web apps is broken access control. The app checks are you logged in but forgets to check is this yours. So /api/v1/invoice?id=1004 happily returns invoice 1004 even when you are logged in as a different user. This is IDOR — Insecure Direct Object Reference.

Try it

You are logged in as alice and you own invoice #1004. Change the id and watch what the server hands back. Then turn on the ownership check to see the fix.

GET /api/v1/invoice?id=

Nothing here is "hacking" in the movie sense — you changed a number in a URL. The vulnerability is the absence of a check, which is exactly why automated scanners miss it and humans find it. The fix is an ownership check on every object access: does this session own the thing it is asking for?

Where to look

  • Numeric or guessable IDs in URLs, forms, and API calls
  • Horizontal access — reaching another user's data at the same privilege level
  • Vertical access — a normal user reaching admin-only functions
  • Hidden admin endpoints that rely on "nobody knows the URL"

Prove it with one cross-account read and stop — do not enumerate the whole table. One invoice that is not yours is a finding; scraping every customer's data is an incident.
