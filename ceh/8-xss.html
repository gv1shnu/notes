Cross-site scripting (XSS)

Difficulty: Hard

XSS is input that becomes script in another user's browser. If an app takes your input and writes it into a page without encoding, a payload like <script>alert(1)</script> runs as code instead of showing as text. From there an attacker can steal sessions, rewrite the page, or act as the victim.

Data vs code

A search box echoes your term back into the results page. Type into it and watch what the server builds — and whether the browser treats it as text or executes it. Then turn on output encoding, the fix.

With encoding on, < and > become &lt; and &gt;, so the browser shows the payload as text and never runs it. Context matters: the same input is dangerous in an HTML body, an attribute, or a JavaScript string in different ways, so the report should name where the reflection landed.

The three kinds, and safe proof

  • Reflected — bounced straight back in the response (like the search box above)
  • Stored — saved and served to other users later; the dangerous one
  • DOM-based — client-side JavaScript does the unsafe write

alert(1) is the industry's "is this thing on?" If the box pops, the point is made — you do not then need to steal anyone's cookies to prove cookies are stealable. Remove any stored markers you leave behind.

Decision tree — context and kind

Two questions shape every XSS: where the input lands (which picks the payload) and how it comes back (which names the kind): 

My input appears on the page
│
├─ WHERE does it land?  (context picks the payload)
│     ├─ HTML body      → <script>… / <img src=x onerror=…>
│     ├─ HTML attribute → break out:  ">  then a tag
│     └─ JS string      → close the quote / ;  then code
│
└─ HOW does it come back?  (this is the "kind")
      ├─ bounced straight back in the response → REFLECTED
      ├─ saved, served to OTHER users later    → STORED  (worst)
      └─ client-side JS writes it unsafely     → DOM-based
            → prove with alert(1); remove any stored markers
