Host discovery

Recon gave us names, domains, and IP ranges — guesses about what exists. Scanning is where we touch only the approved targets and turn guesses into facts. The very first fact: which of these addresses are actually alive? No point port-scanning 254 addresses when three respond. Host discovery is asking "are you there?" politely, rate-limited, and with written permission.

Ways to ask

A single ping is not enough, because ICMP is often blocked. So discovery combines several probes and treats a reply on any of them as "host is up."

Method How it asks Best for
ICMP echoThe classic pingNetworks that allow ICMP
TCP SYN/ACKA packet to a common port (80, 443)When ICMP is filtered
ARPLayer-2 "who has this IP?"Local networks — reliable and quiet
DNS resolutionName resolves to an address in scopeConfirming a target exists at all

The sweep

A "ping sweep" checks a whole range for live hosts without scanning ports. In nmap that is -sn: 

$ sudo nmap -sn 10.10.10.0/24

Nmap scan report for 10.10.10.5
Host is up (0.019s latency).
Nmap scan report for 10.10.10.12
Host is up (0.021s latency).
Nmap scan report for 10.10.10.40
Host is up (0.022s latency).
Nmap done: 256 IP addresses (3 hosts up) scanned in 3.20 seconds

Three live hosts out of 256 — now the noisy port scanning only has to hit those three.

When the host hides

Sometimes a host is up but ignores every discovery probe (a firewall dropping ICMP and unsolicited SYNs). If you already have good reason to believe it exists — recon said so, DNS resolves — tell nmap to skip discovery and just scan it, with -Pn: 

$ sudo nmap -Pn -p 80,443 10.10.10.99

-Pn means "assume the host is up, don't try to prove it first." Useful when discovery is filtered, but slower and noisier, so reach for it deliberately rather than by default. The key mindset from here on: silence is not proof of absence. Firewalls, load balancers, and source-IP rules can all make a live host look dead.
