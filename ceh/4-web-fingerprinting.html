Web fingerprinting

Difficulty: Medium

For a live web host, enumeration starts by working out what the stack is — not to exploit it yet, but to decide where to spend attention. A Laravel app behind nginx invites different tests than a static site on IIS. Fingerprinting is read-only and safe to run broadly.

Read the headers

The response headers alone reveal the server, the framework, cookie behaviour, and which security headers are missing: 

$ curl -sI https://shop.lab.example.com
HTTP/2 200
server: nginx/1.18.0
x-powered-by: Express
set-cookie: session=eyJ...; Path=/
content-type: text/html; charset=utf-8

Three findings already: an Express (Node.js) backend leaked by x-powered-by, a session cookie with no HttpOnly, Secure, or SameSite flags, and no Content-Security-Policy or Strict-Transport-Security in sight.

Fingerprint the whole stack

whatweb rolls many signals into one line: 

$ whatweb https://shop.lab.example.com
https://shop.lab.example.com [200 OK] Country[US], HTTPParty, nginx[1.18.0],
  X-Powered-By[Express], Script[jQuery], Cookies[session]
Signal What it whispers
Response headersServer, framework, cookie flags, missing protections
Favicon hashA Shodan/Censys pivot — same hash often means same known app
TLS cert & SANsMore hostnames, internal naming, expiry / self-signed clues
404 & error pagesStack traces, framework defaults, debug mode left on

Decide what the stack is before pointing any scanner at it. A vulnerability scanner aimed at the wrong stack just produces confident nonsense.
