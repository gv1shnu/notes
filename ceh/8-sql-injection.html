SQL injection

Difficulty: Hard

SQL injection happens when user input becomes part of a database query instead of staying data. When an app builds SQL by gluing strings together — "... WHERE user='" + input + "'" — your quote escapes the string and the rest of your input becomes logic the developer never intended. The result can be authentication bypass, data disclosure, or data modification.

Watch the query collapse

The login below runs SELECT * FROM users WHERE user='{u}' AND pass='{p}' by string concatenation. Your input is really tokenised and evaluated — including -- comments and OR logic. Try a preset, then flip on the parameterised query to watch the same payload fail.

With the parameterised query on, the input is sent to the database as data and never parsed as SQL, so every payload becomes a (wrong) username. That toggle is the entire fix — which is why "use prepared statements" is the only sentence a remediation section really needs here.

Prove it safely

Prove SQL injection with harmless evidence first: a controlled boolean difference (a payload that is true returns a different response length than one that is false), a safe timing check, or a database error message. Do not dump tables unless the scope explicitly asks for data-exposure proof — and never point an automated tool at --dump to do it for you.
