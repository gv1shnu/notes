Authentication and session

Most real-world web compromise begins at authentication, and it resists automation because so much of it is logic, not payloads. Before poking, map the whole identity surface: login, registration, password reset, MFA, "remember me", and logout. Then check how the session behaves across all of them.

Check What good looks like
Cookie flagsHttpOnly, Secure, SameSite all set (see session hijacking)
Session rotationA fresh token on login; the pre-login token stops working
LogoutToken invalidated server-side, not just dropped by the browser
Reset flowSingle-use, expiring, unguessable tokens; no user enumeration
LockoutThrottling exists — but don't be the one who trips it on real users

Reset flows leak

Password reset is where developers keep their most creative mistakes: tokens in the URL, tokens that never expire, and the immortal "we emailed the reset link… and also the new password… in plaintext." Test whether the token is guessable, whether it expires, and whether the reset response quietly reveals which email addresses are registered.

The lockout trap

Credential testing and password spraying are active and can lock out real users — a self-inflicted denial of service. Confirm test accounts and the lockout policy in writing, then spray one password across many users with long delays, never many passwords at one user.
