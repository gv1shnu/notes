Cloud

Difficulty: Hard

Cloud is now the primary external surface for many targets, and cloud findings rarely come from exploits — they come from identity and configuration. Public storage, weak IAM, exposed keys, permissive security groups, missing logging, and open databases are the bread and butter. The single most common one is a storage bucket set to public.

How public is that bucket?

A bucket's access policy decides what an anonymous attacker can do with no credentials at all. Set the policy and run the two commands an attacker would try — list the contents, and download a file.

Bucket policy:

"Public read" already leaks any file whose name is guessed or leaked. "Public list" is worse — the attacker can enumerate every object and then download all of them. This is exactly how customer data ends up on the news, with no exploit involved.

Beyond storage

  • IAM — over-permissioned roles, especially anything with AdministratorAccess
  • Exposed access keys committed to code or leaked in the app
  • Security groups open to 0.0.0.0/0 on databases and admin ports
  • Missing CloudTrail / audit logging
  • Unmanaged secrets and vulnerable machine images

Anonymous, read-only checks (like the public-bucket test) are safe to run. Anything using credentials touches a live account — where a mistake means real cost, state change, or lockout — so it stays under tight scope.

Decision tree — identity and config, not exploits

Cloud findings come from misconfiguration. Run the anonymous, read-only checks freely; anything using credentials touches a live account: 

Cloud target — check config, not exploits
│
├─ Storage bucket → how public?   (anonymous = safe to test)
│     ├─ private            → nothing leaks  ✓
│     ├─ public READ        → any known/guessed filename downloads
│     └─ public LIST + READ → enumerate ALL objects → mass leak
│
├─ IAM → over-permissioned roles (AdministratorAccess)?
├─ Access keys leaked in code / the app?
├─ Security groups open 0.0.0.0/0 on DB / admin ports?
└─ CloudTrail / audit logging missing?

Read-only checks: safe.  Anything using creds touches a live
account (cost, state change, lockout) → keep it tightly scoped.
