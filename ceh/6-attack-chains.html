Attack chains

Difficulty: Hard

Real compromise is rarely one big exploit. It is a chain of small, individually-boring steps where each one unlocks the next. No single link is dramatic; the path is. Understanding chaining is what separates "I found some issues" from "here is how an attacker reaches your crown jewels."

Walk a chain

Step through a typical chain from an anonymous position on the network to root on a server. Watch how the access level climbs — none to user to root — one modest misconfiguration at a time.

Notice that none of those steps was a zero-day. A readable share, a password left in a config file, credential reuse, and one sudo rule — each is a shrug on its own, and together they are a full compromise. That is why the report ranks the chain, not just the isolated findings.

Post-exploitation, kept sane

Once you have access, the questions are: what can this identity reach, what sensitive data is exposed, what systems can it influence, and what would a real attacker do next? Answer them, document the boundary, and stop. Reaching root is the proof; ransacking the machine is not part of the job.

Safe validation, per finding

Each finding type has a minimal proof and a stop condition, decided before you act:

Finding Minimal proof Stop condition
SQL injectionBoolean diff or a single-row selectNo data dump
Default credsOne login screenshotNo password change
Privilege escalationRead one protected file / show the shellNo persistence
SSRFA callback to a listener you controlNo cloud-account pivot

(In the campaign tracker, a helper turns each candidate into a plan like this — proof method, stop condition, cleanup — but a human runs the actual proof. The plan is bookkeeping; the judgement is yours.)
