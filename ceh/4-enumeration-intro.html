Enumeration

Difficulty: Easy

Scanning told us a service is there. Enumeration asks it to talk. Scanning says "SMB is open on 445." Enumeration asks "which shares, users, groups, policies, and permissions will that SMB service reveal?" It is the same target, questioned more closely.

Input Output
Open services and versions from scanning Users, groups, shares, domains, naming patterns
Protocol hints (SMB, LDAP, SMTP…) Service configuration clues and auth methods
Any approved test accounts Misconfiguration candidates for the next phase

The line you do not cross yet

Enumeration is the phase where read-only ends and state-change begins, so it deserves a rule of its own: enumeration should not mutate the target. Reading a banner, listing a share, querying a directory — fine. But if a tool writes a file, changes a setting, or triggers account lockouts, it needs explicit approval first. Credential-based enumeration in particular can lock out real users, which is a denial of service you inflicted on yourself.

External vs internal

Where you are standing decides what you can enumerate. From the outside, the reachable surface is mostly web (80/443), mail (25), and DNS (53). The rich internal services — SMB, LDAP, Active Directory — usually only open up after you have a foothold inside the network (the "assume breach" position). This chapter covers both: the internal identity services first, then the external web enumeration you can do from anywhere.
