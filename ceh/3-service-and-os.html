Service and OS detection

Difficulty: Medium

A port number is only a hint. Port 80 is usually a web server and port 22 is usually SSH, but "usually" does not go in a report. Service detection confirms what is actually running by reading banners and protocol behaviour. This is where 8080 stops being "some random web thing" and becomes evidence.

Version detection

-sV probes each open port and reports the product and version: 

$ sudo nmap -sV -p 22,80,443 10.10.10.5

PORT    STATE SERVICE  VERSION
22/tcp  open  ssh      OpenSSH 8.2p1 Ubuntu 4ubuntu0.5 (Ubuntu Linux; protocol 2.0)
80/tcp  open  http     nginx 1.18.0 (Ubuntu)
443/tcp open  ssl/http nginx 1.18.0 (Ubuntu)

Now you have real leads: an nginx 1.18.0 web server on Ubuntu, and an OpenSSH version you can check against known issues later. Each open service tells its own story:

  • Service name and product, with a version string if exposed
  • The TLS certificate's subject and SANs (which leak more hostnames)
  • HTTP title, Server header, redirects, and status codes
  • Protocol-specific banners: SMB, FTP, SMTP, RDP, database greetings

Safe NSE scripts

Nmap's scripting engine adds targeted checks. The default set (-sC) is safe; some categories are not. Start safe: 

$ sudo nmap -sV -sC -p 443 10.10.10.5
| ssl-cert: Subject: commonName=shop.lab.example.com
| Subject Alternative Name: DNS:shop.lab.example.com, DNS:api.lab.example.com
| ssl-enum-ciphers:
|   TLSv1.2: (weak ciphers found)

That certificate just handed you another subdomain (api.) and a TLS weakness to note. Categories matter: -sC runs "safe" and "default" scripts, but --script vuln or the intrusive and exploit categories actively poke at services and can break them — never the automatic choice.

OS detection

-O guesses the operating system from TTL values, TCP window sizes, and how the stack behaves. It is a guess, not gospel: 

$ sudo nmap -O 10.10.10.5
Running: Linux 5.X
OS details: Linux 5.4 - 5.15
Network Distance: 2 hops

Open-port patterns and banners refine the guess: Windows, Linux, a network appliance, a printer, a camera, or a cloud load balancer each leave different fingerprints.

The edge is not the origin

On an internet-facing target you are usually scanning a CDN/WAF edge, not the real server, so the versions and headers may belong to Cloudflare rather than the app. If the engagement needs the true origin, origin discovery is its own small quest: historical DNS, leaked SPF/MX records pointing at direct-connect hosts, misconfigured subdomains that resolve straight through, and later, SSRF from the app itself. Until you have proven otherwise, label a result edge — half of all "the server behaved weirdly" notes are just the CDN.

Raw vs interpretation

Keep the two apart. Raw says "443 open, nginx 1.18." Interpretation says "web app candidate — check TLS, headers, auth, and tech stack next." And always record a confidence level, so a soft guess never becomes a loud claim in the report.

Decision tree — route each service to its next move

Version detection turns open ports into leads. This is the router that sends each confirmed service to the right enumeration page — run sudo nmap -sV -sC first, then branch: 

Open port confirmed → sudo nmap -sV -sC
│
├─ 80 / 443 / 8080  (http, ssl/http)
│     → web fingerprint + content discovery (whatweb, ffuf)
│
├─ 445 / 139  (SMB)          → enum4linux-ng, smbclient, nxc
├─ 389 / 636  (LDAP)         → ldapsearch, BloodHound
├─ 22 (SSH) / 21 (FTP)       → banner → check version for CVEs
├─ 25 (SMTP) / 53 (DNS) / 161 (SNMP) → mail-dns-snmp checks
├─ 3306 / 5432 / 1433  (DB)  → greeting, default creds
│
└─ Want NSE scripts?
      ├─ -sC  (safe / default)        → always fine
      └─ --script vuln / intrusive    → can break things → get scope
