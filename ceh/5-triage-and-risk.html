Triage and risk

Difficulty: Medium

A scanner produces a long, confident list. Your job is not to believe it — it is to prove what is real, drop what is not, and rank what remains by how much it actually matters here. Automated tools generate candidates; humans make findings.

False positives first

Before ranking anything, delete the noise. Version-string matching is especially unreliable: a backported patch hides behind an old banner, so "OpenSSH 7.x, therefore vulnerable" is a guess, not a finding. Typical false positives:

  • An "expired certificate" flagged on a CDN edge that is not the real endpoint
  • A CVE matched purely on a version banner, with no proof the vulnerable code path exists
  • Informational findings dressed up as issues

The boring settings are where the real findings usually live anyway: default credentials, anonymous access, old TLS, missing MFA, exposed admin panels, world-readable storage, debug mode left on. Tool-heavy courses over-index on scanners; real reports are full of misconfigurations.

Risk is not just CVSS

A CVSS score describes a vulnerability in the abstract. Risk describes it in this environment. A medium issue on a public payment app can matter more than a high issue buried in a sealed lab VLAN. Rank with context:

Question Why it changes the ranking
Is it reachable?An unreachable bug is lower priority unless a path exists to it.
Is authentication required?Unauthenticated issues usually rank higher.
Is exploitation reliable?Speculative findings need softer wording and lower rank.
What is the blast radius?One test box versus domain-wide compromise are different planets.
Is it on the KEV list?Known-exploited issues jump the queue.

Plan safe validation

For each surviving candidate, decide the least risky way to prove it in the next phase. Often a screenshot, a version fact, a harmless request, or a configuration excerpt is enough. Every validation step should carry a stop condition set in advance: if the next action could change data, disrupt service, or expose secrets, pause and confirm scope. You do not need to burn down the kitchen to prove the stove is on.

The output of this chapter is a ranked list of confirmed-or-plausible weaknesses, each with a validation plan — the input to exploitation.
