Recon frameworks

Difficulty: Medium

Doing each lookup by hand is fine for a few hosts. When there are dozens, a framework keeps the results organised and repeatable. Recon-ng is the classic one: a web-reconnaissance framework with independent modules and a built-in database, styled after Metasploit's console.

Module categories

Modules are grouped in a marketplace by what they do:

  • discovery — find new things (interesting files, related hosts)
  • recon — the bulk: domains → hosts, hosts → ports, contacts, credentials
  • import — bring in your own lists
  • exploitation — light interaction with found services
  • reporting — export the database to CSV, HTML, and so on

A workspace, start to finish

Everything lives in a named workspace, and results accumulate in tables you can query. A minimal end-to-end run: 

$ recon-ng
[recon-ng][default] > marketplace install all
[recon-ng][default] > workspaces create acme
[recon-ng][acme] > db insert domains
domain (TEXT) > example.com

[recon-ng][acme] > modules load recon/domains-hosts/hackertarget
[recon-ng][acme][hackertarget] > run
[*] example.com => 34 hosts found

[recon-ng][acme][hackertarget] > back
[recon-ng][acme] > show hosts
[recon-ng][acme] > db export csv recon_output.csv

Load a module, point it at the domain, run it, and the discovered hosts land in the hosts table. Chain a few modules and the database fills with subdomains, IPs, and contacts — then export it as the input for the scanning phase.

The handoff is the point: whatever hosts end up in the database, filtered to what is in scope, become the target list you scan next.
