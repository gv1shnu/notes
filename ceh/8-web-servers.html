Web servers

Web is the dominant surface in a black-box test, and web testing flows in one direction: from the server's exposure, to the application's behaviour, to the data behind it. This chapter walks that path. It starts with the server itself — the software serving the pages, before any application logic.

What to check

  • Server and version exposure (the Server header)
  • Default pages and sample apps left installed
  • Directory listing turned on
  • TLS configuration and certificates
  • Dangerous HTTP methods enabled
  • Backup and config files under the web root
  • Reverse-proxy and CDN behaviour

Dangerous methods

Ask the server which HTTP methods it allows. PUT or DELETE on a content path can mean you upload or remove files directly: 

$ curl -sI -X OPTIONS https://shop.lab.example.com/
HTTP/2 200
allow: GET, POST, PUT, DELETE, OPTIONS

PUT and DELETE being listed is worth a careful, in-scope test — writing a harmless marker file proves it without damage.

TLS configuration

testssl.sh or sslscan grade the TLS setup — protocol versions, weak ciphers, cert problems: 

$ testssl.sh --quiet https://shop.lab.example.com
 TLS 1.0    offered (deprecated)
 TLS 1.3    offered (OK)
 Certificate Validity  expires in 12 days (!)

Deprecated TLS 1.0 and a certificate about to expire are both reportable. Also probe for leftover files — /backup.zip, /.git/, /config.php.bak — which spill source and secrets more often than any clever exploit.
