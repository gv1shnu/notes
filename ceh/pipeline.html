Cross-cutting / Automation

Pentest pipeline automation

The phase pages teach what to do manually. This page describes how to wire phases together with Python so recon output becomes scan input, scan output becomes enum input, and so on — with human checkpoints where judgment is required.

Automate the clipboard, not the conscience. The pipeline collects and routes evidence; you still decide what to validate, what is in scope, and what goes in the report.

What automation is good for here

Automate Keep manual
Subdomain collection, DNS resolve, dedupe Deciding which hosts are worth deeper testing
Batch nmap from an inventory file Interpreting weird firewall / CDN behaviour
Routing open ports → enum command checklists Running invasive enum (lockouts, writes, sprays)
CVE lookup from version strings False-positive triage and business risk ranking
Header crawls, TLS audits, report assembly Exploitation, privesc, social engineering
JSONL campaign timeline Final severity wording and client delivery

Scoring automation: the four-axis model

The table above is a verdict; this is the method behind it. Score any action — any tool, any script — on four axes. The last axis is the one black-box work keeps forgetting: from the outside you cannot see internal impact, so you can never fully bound the blast radius of what you automate.

AxisSafe endScary end
ContactPassive (third-party data only)Active (your packets hit the target)
EffectRead-only / idempotentState-changing / irreversible
Fan-outOne attributed assetMany discovered/guessed assets
VisibilityYou can see the consequenceYou cannot (black-box default)

Those axes collapse into four tiers. Tag every phase script with exactly one:

  • TIER 1  Automate freely — passive + read-only.
  • TIER 2  Automate behind a runtime scope gate + rate cap — active + read-only.
  • TIER 3  Automate to propose, execute by hand — state-changing or wide fan-out.
  • TIER 4  Never automate — irreversible + blind blast radius.
The pipeline's job is to do all the Tier 1 and Tier 2 grunt work tirelessly, and to stop with its hands visible at Tier 3. Anything Tier 4 doesn't get a script; it gets a human and a written stop condition.

Simulation: the automation tier classifier

Set the four axes for an action and see which tier it lands in — or hit a preset to see how real tools score. This is the decision you make, consciously or not, every time you reach for a script.

Architecture

One orchestrator (scripts/pipeline.py) runs phase scripts in order. Each phase reads a JSON artifact from the previous step and writes its own. Scope checks and logging wrap every step. 

scope.json
    │
    ▼
┌─────────────┐     recon/inventory.json
│    recon    │ ─────────────────────────►
└─────────────┘
    │  GATE: human reviews inventory
    ▼
┌─────────────┐     scans/summary.json
│   scanning  │ ─────────────────────────►
└─────────────┘
    │  GATE: confirm scan intensity / targets
    ▼
┌─────────────┐     enum/details.json
│ enumeration │ ── (+ enum/checklists/*.md dry-run)
└─────────────┘
    │  parallel: web_recon on :80/:443, tls_cert_audit
    ▼
┌─────────────┐     vuln/candidates.json
│ vuln analysis│ ────────────────────────►
└─────────────┘
    │  GATE: triage — drop FPs, rank risk
    ▼
┌─────────────┐     validate/plans.json
│  validation │ ── (plans only; proof is manual)
└─────────────┘
    │  GATE: human runs approved proofs in lab
    ▼
┌─────────────┐     findings.json
│  reporting  │ ──► reports/draft.html
└─────────────┘

Specialty pages (network, web, platform) plug in as parallel branches after scanning or enumeration — not a separate linear step after exploitation.

Engagement data layout

All machine output lives under data/<engagement-id>/ so notes stay separate from lab artifacts. Copy scope.json.example to data/<engagement-id>/scope.json.

Artifact contracts

Each phase page lists its pipeline I/O file. Schemas are intentionally small JSON — extend fields as you implement scripts.

File Produced by Minimum shape
recon/inventory.json subdomain_enum.py { "domain", "hosts": [{ "host", "ips", "sources" }] }
scans/summary.json port_scan_batch.py { "hosts": [{ "host", "ports": [{ "port", "proto", "service", "version" }] }] }
enum/details.json service_enum.py { "hosts": [{ "host", "services": [{ "port", "findings": [] }] }] }
vuln/candidates.json cve_lookup.py { "candidates": [{ "host", "issue", "cves", "validation_plan" }] }
validate/plans.json validation_plan.py { "plans": [{ "id", "proof_method", "stop_condition", "cleanup" }] }
findings.json finding_add.py (manual CLI after proof) { "findings": [{ "title", "severity", "evidence", "impact", "fix" }] }

Human gates (checkpoints)

The pipeline pauses after phases marked gate: true until you approve in checkpoints.json or pass --force in a lab-only run.

  1. After recon Review inventory — drop out-of-scope hosts, mark scan priorities.
  2. After scanning Confirm scan covered intended targets; note CDN/firewall blind spots.
  3. After vuln analysis Triage candidates — delete false positives before any validation talk.
  4. Before report Only confirmed findings via finding_add.py; run redact_secrets.py.

Running the pipeline

# From ceh/
pip install -r scripts/requirements.txt

# Init engagement folder from example scope
python scripts/pipeline.py init --engagement lab-2025-06 --scope scope.json.example

# Dry-run: show planned steps, no subprocess calls
python scripts/pipeline.py run --engagement lab-2025-06 --dry-run

# Run one phase
python scripts/pipeline.py run --engagement lab-2025-06 --phase recon

# Run all automated phases (stops at gates)
python scripts/pipeline.py run --engagement lab-2025-06

# Approve a gate after review
python scripts/pipeline.py approve --engagement lab-2025-06 --gate post_recon

# Lab only: skip gates
python scripts/pipeline.py run --engagement lab-2025-06 --force
