Methodology

A pentest reads like a campaign, not a bag of random commands. Each phase consumes the previous phase's output and produces the next phase's input. Recon says "I found doors." Scanning says "I checked the locks." Exploitation says "one lock is made of noodles." Reporting says "please buy a real lock." In that order.

The phases

  1. Rules of engagement — get written scope, target list, time window, allowed techniques, and reporting format. Everything downstream depends on this.
  2. Reconnaissance — collect public facts: domains, subdomains, IP ranges, people, technologies, exposed services.
  3. Scanning — touch approved systems carefully to map live hosts, ports, and service versions.
  4. Enumeration — go deeper on each service: users, shares, configs, identity surfaces.
  5. Vulnerability analysis — turn raw findings into a ranked shortlist of likely weaknesses, minus false positives.
  6. Exploitation — validate impact inside scope. Prove the issue with the smallest blast radius, do not cosplay as a disaster movie.
  7. Network, web, and platform checks — the specialty branches: sniffing, web apps, wireless, cloud, and so on. They run in parallel, not strictly after exploitation.
  8. Reporting and cleanup — deliver evidence, risk, reproduction, and fixes; remove test artifacts; confirm a retest.

Written as a pipeline, the whole campaign is one line: 

scope -> recon -> scan -> enumerate -> analyze -> validate -> report -> retest

The campaign tracker

Because each phase hands a file to the next, the handoffs can be automated. These notes ship a small set of Python scripts (see the tracker) that route each phase's output into the next and log a timeline, with human approval gates between phases. Treat it as bookkeeping that keeps the campaign straight — handy, but never the point. The thinking, the scope calls, and the actual testing stay yours. The phase pages that follow teach the manual work first; the tracker just remembers it.
