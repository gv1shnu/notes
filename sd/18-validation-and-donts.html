Validating the design

Difficulty: Hard

18. Call chains & design don'ts.

Validate with call-chain diagrams: for each use case, draw how the client, managers, engines, and resource-access interact (solid arrows = synchronous, dashed = queued). If every use case has a valid call chain, the design holds water — before a line of code. (Sequence diagrams add time/order but are slower to draw.)

Then scrub against the design don'ts. Each one, if violated, means functional decomposition is hiding — go find it.

Don't Because
Client calls multiple managers in one use case the client is stitching them → functional managers
Manager queues calls to >1 manager law of 0/1/∞ — "one" really means "all", so use Pub/Sub
Queue a call to an engine or resource access they're utilitarian; they have no life of their own in isolation
Engines / clients / resources publish events publishing is a managerial decision (state changed)
Engines subscribe to events a subscription triggers a use case → that's a manager/client job
Engines call engines; resource-access call each other sideways = coupling; and a real atomic verb never needs another

That's the Method end to end: stop designing against requirements, decompose by volatility, lay it out as Clients → Managers/Engines → ResourceAccess → Resources, and compose the few core use cases from ~10 reusable blocks. Then change becomes cheap — which is the whole point.
