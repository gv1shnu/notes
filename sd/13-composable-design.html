Composing the system

13. Composable design.

Most use cases are variations of a few. Split them:

Core use cases Fluff
the essence of the business — "bring the customer to the door" everything else (variations, edge cases)
very few: usually 1–3, never >6 near-infinite

Your mission: find the smallest set of building blocks you can compose to satisfy each core use case. Because all other use cases are just different interactions of those same blocks — a change in requirements becomes a change in composition, not in decomposition. The design doesn't move.

That smallest set lands at about ~10 components (a couple of clients, managers, engines, a few resource-accesses/resources, half a dozen utilities). 10! combinations is enough for anything. Same as a car (~10 parts), a laptop (~10), a body (~10 organs). Humans run on the same components as a 160,000-year-old hunter-gatherer — one core use case: survive; everything else is composition.

The universal rule that follows: features are aspects of integration, not implementation. No single part holds "the feature" — it emerges from integrating parts (true all the way down to the screw).
