We've been working with one table at a time.
Every query so far — SELECT, WHERE, GROUP BY, aggregates — touched a single table.

But real databases never store everything in one table. That's the whole point of the introduction: splitting data across tables eliminates redundancy.

The problem now is: how do we query data that lives in multiple tables?

The Problem — Data is Split

In PUBG, a match involves players. Each player belongs to a squad. If we stored everything in one table, it would look like this:

player_id username kills squad_id squad_name squad_region
1 Clocky 8 101 Team Liquid EU
2 Grongat 5 101 Team Liquid EU
3 Mitr0 12 101 Team Liquid EU
4 Ubah 3 101 Team Liquid EU
5 Pio 7 102 Natus Vincere EU
6 Esportsnation 9 102 Natus Vincere EU

Problem: squad_name and squad_region are repeated for every single player on that squad. If Team Liquid changes their region, you'd have to update four rows — and risk inconsistency if you miss one. This is exactly the redundancy problem from the introduction.

Solution: Split into two tables. Store squad data once. Reference it from the players table using a foreign key.

Two Tables — players and squads

squads

squad_id squad_name region
101 Team Liquid EU
102 Natus Vincere EU
103 Gen.G AS

players

player_id username squad_id kills
1 Clocky 101 8
2 Grongat 101 5
3 Mitr0 101 12
4 Ubah 101 3
5 Pio 102 7
6 Esportsnation 102 9
7 FaZe_suk NULL 2

squad_id in the players table is a foreign key — it references squad_id in squads. Note that player FaZe_suk has no squad_id (NULL). He's a solo player. We'll see why this matters soon. Also note Gen.G (squad 103) has no players yet in our table.

The redundancy is gone. But now there's a new problem: if you want to see each player's username alongside their squad name, the data is in two separate tables. How do you combine them in one query?

JOINs — The Solution

A JOIN combines rows from two tables based on a related column — usually a foreign key. The result is a temporary combined table that exists only for the duration of your query.

The column we join on is the link between the two tables. Here, that link is squad_id.

INNER JOIN

Returns only the rows where there is a match in both tables.

Question: Show each player's username and their squad name. 



SELECT players.username, squads.squad_name
FROM players
INNER JOIN squads ON players.squad_id = squads.squad_id;

Result:

username squad_name
Clocky Team Liquid
Grongat Team Liquid
Mitr0 Team Liquid
Ubah Team Liquid
Pio Natus Vincere
Esportsnation Natus Vincere

Notice: FaZe_suk is missing. His squad_id is NULL — there's no match in squads. INNER JOIN discards any row that doesn't have a match on both sides.

Gen.G is also missing. It exists in squads, but has no matching players. Same reason.

LEFT JOIN

Returns all rows from the left table, plus matching rows from the right table. If there's no match on the right side, the result shows NULL for those columns.

Question: Show all players, even those without a squad. 



SELECT players.username, squads.squad_name
FROM players
LEFT JOIN squads ON players.squad_id = squads.squad_id;

Result:

username squad_name
Clocky Team Liquid
Grongat Team Liquid
Mitr0 Team Liquid
Ubah Team Liquid
Pio Natus Vincere
Esportsnation Natus Vincere
FaZe_suk NULL

FaZe_suk appears now — with NULL for squad_name because he has no squad. The left table is players. Every player is preserved.

Gen.G is still missing. The right table (squads) is not fully preserved in a LEFT JOIN.

RIGHT JOIN

Returns all rows from the right table, plus matching rows from the left table. If there's no match on the left side, the result shows NULL for those columns.

Question: Show all squads, even those with no players registered yet. 



SELECT players.username, squads.squad_name
FROM players
RIGHT JOIN squads ON players.squad_id = squads.squad_id;

Result:

username squad_name
Clocky Team Liquid
Grongat Team Liquid
Mitr0 Team Liquid
Ubah Team Liquid
Pio Natus Vincere
Esportsnation Natus Vincere
NULL Gen.G

Gen.G now appears — with NULL for username because no players are linked to them yet. The right table is squads. Every squad is preserved.

FaZe_suk is gone again. He exists in the left table (players) with no match in squads. RIGHT JOIN does not preserve the left table.

Note: Usually developers don't use right join often, and just switch the tables and use left join as usual. Right join was added to the features of SQL for the sake of completeness.

FULL OUTER JOIN

Returns all rows from both tables. Where there's no match, NULL fills the gap on whichever side is missing.

Question: Show all players and all squads — matched where possible, NULL where not. 


MySQL — not supported natively. Simulate with UNION:


SELECT players.username, squads.squad_name
FROM players
LEFT JOIN squads ON players.squad_id = squads.squad_id

UNION

SELECT players.username, squads.squad_name
FROM players
RIGHT JOIN squads ON players.squad_id = squads.squad_id;




PostgreSQL


SELECT players.username, squads.squad_name
FROM players
FULL OUTER JOIN squads ON players.squad_id = squads.squad_id;

Result:

username squad_name
Clocky Team Liquid
Grongat Team Liquid
Mitr0 Team Liquid
Ubah Team Liquid
Pio Natus Vincere
Esportsnation Natus Vincere
FaZe_suk NULL
NULL Gen.G

Everyone appears. FaZe_suk (no squad) and Gen.G (no players) both show up with NULL on the missing side.

Summary — Which JOIN to use?

JOIN Type Returns Use when
INNER JOIN Only matching rows from both tables You only want data that is fully linked on both sides
LEFT JOIN All rows from left + matches from right You want to keep all records from the left table, even unmatched ones
RIGHT JOIN All rows from right + matches from left You want to keep all records from the right table, even unmatched ones
FULL OUTER JOIN All rows from both tables You want a complete picture — no row from either side should be dropped

Using Aliases to Write Cleaner JOINs

Writing players.username and squads.squad_name repeatedly gets verbose. Aliases let you shorten table names for the duration of the query. 



SELECT p.username, s.squad_name
FROM players p
INNER JOIN squads s ON p.squad_id = s.squad_id;

p is now shorthand for players. s is shorthand for squads. The query does the same thing — it just reads faster.

JOINs with WHERE, ORDER BY, and Aggregates

A JOIN gives you a combined table. Everything you already know — WHERE, ORDER BY, GROUP BY, aggregate functions — works on top of it.

Question: Which squad has the highest total kills? 



SELECT s.squad_name, SUM(p.kills) AS total_kills
FROM players p
INNER JOIN squads s ON p.squad_id = s.squad_id
GROUP BY s.squad_name
ORDER BY total_kills DESC;

Result:

squad_name total_kills
Team Liquid 28
Natus Vincere 16

The query joins the two tables first, then groups the combined result by squad, then sums kills per group, then orders by total. This is the execution order you studied in the SQL execution flow chapter.

Question: Show players from the EU region only. 



SELECT p.username, s.squad_name, s.region
FROM players p
INNER JOIN squads s ON p.squad_id = s.squad_id
WHERE s.region = 'EU';

Result:

username squad_name region
Clocky Team Liquid EU
Grongat Team Liquid EU
Mitr0 Team Liquid EU
Ubah Team Liquid EU
Pio Natus Vincere EU
Esportsnation Natus Vincere EU

The WHERE filter runs after the JOIN. The combined table is built first, then filtered.

Note: Joining more than two tables is possible — just chain additional JOIN clauses. Each one links the new table to the running result using a shared column.
