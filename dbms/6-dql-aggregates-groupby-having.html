Aggregate Functions

Aggregate functions perform calculations on multiple rows. You can always fetch the complete data and write custom aggregate functions on that data. However, there are two problems that come with it. 1. Fetching complete wastes bandwidth and slows the application. 2. Writing custom aggregation function needs manual validation according to the schema. Solution: Let the DBMS do the computation close to the data, return one number instead of a million rows.

Function Purpose
COUNT() Counts rows
AVG() Average value
MAX() Largest value
MIN() Smallest value
SUM() Total value

Current table:

username rank rr
TenZ Radiant 842
aspas Immortal 611
yay Radiant 790

MySQL / PostgreSQL


SELECT AVG(rr)
AS average_rr
FROM players;

AS renames the output column.

Output:

average_rr
747.67

GROUP BY

GROUP BY creates groups of similar rows.

Current table:

username rank rr
TenZ Radiant 842
aspas Immortal 611
yay Radiant 790

AVG(rr) gives you one average across all players. But what if you want the average per rank? You'd have to run a separate query for each rank. With 10 ranks that's 10 queries. With 100 ranks, 100 queries. GROUP BY solves this: one query, one result per group.

MySQL / PostgreSQL


SELECT rank, COUNT(*)
AS total_players
FROM players
GROUP BY rank;

Output:

rank total_players
Radiant 2
Immortal 1

HAVING

HAVING filters grouped data.

Current grouped result:

rank total_players
Radiant 2
Immortal 1

MySQL / PostgreSQL


SELECT rank, COUNT(*)
AS total_players
FROM players
GROUP BY rank
HAVING COUNT(*) > 1;

Output:

rank total_players
Radiant 2

WHERE filters rows before grouping.
HAVING filters groups after grouping.

This may feel unusual if you're coming from languages like C, C++, Java, or Python, where statements typically execute from top to bottom. SQL follows its own logical execution order, which is not the same as the order in which the clauses are written. We'll study the SQL execution flow shortly.
