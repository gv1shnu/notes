SQL Execution Flow (so far)

We have now met every clause of a basic query — SELECT, FROM, WHERE, ORDER BY, LIMIT and OFFSET. Before adding more, it is worth pausing on a subtle point: the order in which we write these clauses is not the order in which the database runs them.

Example Query:

MySQL / PostgreSQL

SELECT username, score
FROM players
WHERE score > 400
ORDER BY score DESC
LIMIT 2 OFFSET 1;

Parsing Order vs Execution Order

SQL is a declarative language: we describe the result we want, not the steps to compute it. SQL has a fixed parsing order (how a query must be written) which is separate from its logical execution order (how the database actually evaluates it). For the clauses covered so far:

Clause Parsing order (how you write it) Execution order (how it runs)
SELECT 1 3
FROM 2 1
WHERE 3 2
ORDER BY 4 4
LIMIT / OFFSET 5 5

SELECT is written first but runs third — the database must load and filter rows before it knows which columns to keep. This single fact explains why a column alias defined in SELECT cannot be used in WHERE: when WHERE runs, SELECT has not happened yet.
