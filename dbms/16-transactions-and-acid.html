Transactions and ACID Properties

A single UPDATE is safe — it either happens or it doesn't. But real operations are rarely a single statement.

Problem: Moving 100 points from TenZ to Zekken is two writes: subtract from one, add to the other. If the server crashes between them, the points have left TenZ and never arrived — 100 points vanish, and the database is in a state your rules say is impossible. 

UPDATE players SET score = score - 100 WHERE username = 'TenZ';
-- crash here, and the points are gone
UPDATE players SET score = score + 100 WHERE username = 'Zekken';

Solution: Wrap the steps in a transaction — a single unit of work that the database treats as indivisible. Either both updates land, or neither does.

Using transactions

PostgreSQL ( START TRANSACTION in MySQL )

BEGIN;
    UPDATE players SET score = score - 100 WHERE username = 'TenZ';
    UPDATE players SET score = score + 100 WHERE username = 'Zekken';
COMMIT;

COMMIT makes every change permanent at once. If anything goes wrong before it, ROLLBACK throws the whole transaction away and the database is exactly as it was before BEGIN — as though the half-finished transfer never happened. 

ROLLBACK;   -- undo everything since BEGIN

ACID

The guarantee a transaction makes has four parts, abbreviated ACID.

PropertyPromise
AtomicityAll steps happen, or none do
ConsistencyThe database moves from one valid state to another
IsolationConcurrent transactions don't see each other's half-done work
DurabilityOnce committed, it survives a crash

Atomicity is what saved the transfer above — the crash triggers a rollback, and the missing 100 points reappear. The database keeps an undo log so it can always walk a failed transaction backwards.

Consistency means every constraint you declared still holds after the transaction. If a CHECK (score >= 0) would be violated, the transaction is refused, not half-applied.

Isolation means a second transaction reading the players table mid-transfer never sees the state where the points have left TenZ but not reached Zekken. How strongly the database enforces this is the subject of the next chapter, concurrency control.

Durability means once COMMIT returns, the data is safe even if the power dies one millisecond later. This is delivered by a write-ahead log: the change is written to a durable log before the commit is acknowledged, so a crash can always replay it. That log is the heart of the recovery chapter.

Partial rollback with SAVEPOINT

Inside a long transaction you can set a savepoint and roll back to it without discarding everything before it. 

SAVEPOINT before_bonus;
    UPDATE players SET score = score + 50 WHERE team_id = 1;
ROLLBACK TO before_bonus;   -- undo just the bonus, keep the rest

One word of caution

The C in ACID is not the C in CAP. ACID's Consistency means "the data obeys its constraints." CAP's Consistency — which you'll meet in the scaling chapter — means "every node in a distributed system shows the same value at the same time." Same word, different promise; mixing them up is a classic trap.

Summary

  • Problem: A multi-step operation can be left half-finished by a crash.
    Solution: A transaction makes the steps atomic — all or nothing — via BEGIN / COMMIT / ROLLBACK.
  • Problem: What exactly does the database promise?
    Solution: ACID — Atomicity, Consistency, Isolation, Durability.
