Window Functions

GROUP BY taught us to summarise. But it summarises by destroying: a query that groups players by region returns one row per region, and the individual players vanish.

We use the familiar players table.

Problem: Show every player alongside their region's average score, so you can see who is above and below it. GROUP BY can give you the averages or the players, but not both in one result — grouping collapses the very rows you want to keep.

Solution: A window function computes a value across a set of related rows without collapsing them. Every input row survives and gains an extra column.

The set of rows it looks at is called the window, declared with OVER.

OVER and PARTITION BY

PARTITION BY is GROUP BY's non-destructive twin. It splits rows into groups for the calculation, then hands every row back. 

SELECT username, region, score,
       AVG(score) OVER (PARTITION BY region) AS region_avg
FROM players;

PARTITION BY splits the rows into these three colored groups purely for the calculation — every row keeps its seat, it just gains its group's average.

Ranking — ROW_NUMBER, RANK, DENSE_RANK

The headline use of windows is ranking. All three need an ORDER BY inside the window to define the order being ranked. 

SELECT username, region, score,
       ROW_NUMBER() OVER (PARTITION BY region ORDER BY score DESC) AS row_num,
       RANK()       OVER (PARTITION BY region ORDER BY score DESC) AS rnk,
       DENSE_RANK() OVER (PARTITION BY region ORDER BY score DESC) AS dense
FROM players;

Watch what happens in EMEA, where Boaster and Derke are tied at 710 — this is the whole reason three ranking functions exist.

  • ROW_NUMBER always gives distinct numbers, so the tie is broken arbitrarily (1, 2).
  • RANK gives ties the same number, then skips — two firsts would be followed by a 3.
  • DENSE_RANK gives ties the same number and doesn't skip — two firsts followed by a 2.

Same tie, three different next-position numbers. "Top 3 by RANK" would silently miss whatever lands on the skipped number.

Choosing wrong is a classic bug: "the top 3" built on ROW_NUMBER silently drops a genuinely tied player.

Running totals and moving views

Add an ORDER BY to an aggregate window and it accumulates as it goes, giving a running total. 

SELECT username, score,
       SUM(score) OVER (ORDER BY score) AS running_rr
FROM players;

Each row's running_rr is the sum of every score up to and including it, in score order. This is the engine behind leaderboards, cumulative revenue, and "balance after each transaction".

LAG and LEAD

These reach to the previous or next row in the window — perfect for comparing a row to its neighbour. 

SELECT username, score,
       LAG(score)  OVER (ORDER BY score DESC) AS rr_above,
       LEAD(score) OVER (ORDER BY score DESC) AS rr_below
FROM players;

For each player you now see the score of the player ranked just above and just below them. Subtract to get the gap to the next player up the ladder. The top row's rr_above is NULL — there's nobody above it.

NTILE

NTILE(n) slices the ordered rows into n roughly equal buckets — useful for percentiles and tiering. 

SELECT username, score,
       NTILE(3) OVER (ORDER BY score DESC) AS tier
FROM players;

The top third land in tier 1, the middle in 2, the bottom in 3.

The pattern: top-N per group

Window functions can't go in WHERE (same execution-order reason aggregates can't — the window is computed at SELECT time, after WHERE). So to keep only the top player per region, you compute the level in a subquery, then filter the result — tying together this page and the subqueries chapter: 

SELECT username, region, score
FROM (
    SELECT username, region, score,
           ROW_NUMBER() OVER (PARTITION BY region ORDER BY score DESC) AS rn
    FROM players
) ranked
WHERE rn = 1;

Optimisation

The window's ORDER BY decides the work: ranking and running totals force the database to sort each partition, so an index on the partitioning and ordering columns can turn a sort into a cheap scan. The frame clause (ROWS BETWEEN ...) lets you control exactly how many surrounding rows a moving calculation sees, instead of the default of "everything up to the current row" — reach for it once running totals and moving averages get heavy. Window functions exist in PostgreSQL and in MySQL 8.0 and later; on older MySQL you're back to correlated subqueries doing the same job, slower.

Summary

  • Problem: GROUP BY summarises by collapsing rows, but you often want the summary and the rows.
    Solution: Window functions compute across a window with OVER, keeping every row.
  • Problem: Ties make "level" ambiguous.
    Solution: ROW_NUMBER, RANK, and DENSE_RANK each handle ties differently — pick on purpose.
