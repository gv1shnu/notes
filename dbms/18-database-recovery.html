Database Recovery

Durability promised that a committed transaction survives a crash, and Atomicity promised an uncommitted one leaves no trace. Recovery is how the database keeps both promises across the worst moment in its life — the instant the power dies.

Problem: A crash can strike at the cruelest time. It might hit mid-transaction, after some changed pages have already been flushed to disk — so the disk now holds half of a transfer that must never half-exist. Or it might hit just after a commit returned but before the changes reached disk — so committed work is about to be lost. In both cases the on-disk state, taken alone, is wrong.

Solution: Never trust the data pages alone. Before any change is considered safe, write a record of it to a durable, append-only log. On restart, the log tells the database exactly what to redo and what to undo to reach a correct state.

Kinds of failure

Write-ahead logging

The single rule the whole system rests on: the log record for a change must reach durable storage before the change itself is allowed to. That's write-ahead logging (WAL). Each log record carries enough to reconstruct the change both ways — a before-image to undo it, an after-image to redo it.

Because the log records the COMMIT, the database can always tell which transactions finished and which didn't — and that single fact drives everything that follows.

Redo and undo

Two operations rebuild a correct state from the log:

  • Redo re-applies the after-images of committed transactions whose changes may not have reached disk — keeping the Durability promise.
  • Undo reverses the before-images of uncommitted transactions whose changes did reach disk — keeping the Atomicity promise.

Both are needed because real engines are greedy: they let a transaction's dirty pages drift to disk before it commits (so undo is needed), and they don't force a transaction's pages to disk at commit time (so redo is needed). The log makes that freedom safe.

Checkpoints

Replaying the entire log from the beginning of time would make restarts unbearable. Every so often the database takes a checkpoint: it flushes outstanding changes and writes a marker recording which transactions were still active. Recovery can then start from the last checkpoint instead of the dawn of the database, replaying only recent history.

Recovery on restart

When the database comes back up it walks the log in three passes, the shape made famous by the ARIES algorithm:

  1. Analysis — read forward from the last checkpoint to find which transactions were in flight and which pages were dirty.
  2. Redo — repeat history, re-applying every logged change so the database matches its state at the moment of the crash.
  3. Undo — roll back the transactions that never committed, leaving only durable, committed work.

When the disk itself dies

WAL handles a crash, but not a melted drive — the log lives on that drive too. Media failure is survived with backups: take a periodic base backup, keep shipping the log, and to recover you restore the backup and replay the log up to any chosen moment. That last ability is point-in-time recovery — rewinding the database to one second before someone ran DELETE without a WHERE.

And one bonus: if you can ship the log to a backup machine to replay, you can ship it to a live second server that replays continuously — which is exactly how replication, in the scaling chapter, is built.

Summary

  • Problem: A crash leaves uncommitted changes on disk and committed changes off it.
    Solution: Write-ahead logging records every change durably first; restart redoes committed work and undoes the rest.
  • Problem: Replaying the whole log is too slow, and a dead disk takes the log with it.
    Solution: Checkpoints bound the replay; backups plus log replay survive media failure and enable point-in-time recovery.
