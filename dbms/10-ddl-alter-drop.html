DDL, continued — ALTER and DROP

Back in the DDL chapter we created tables and deliberately put ALTER and DROP aside, because changing or removing a structure only makes sense once there is data and a schema worth changing. Now there is.

Problem: A schema is never finished. A new column is needed, a type was chosen badly, a table has outlived its purpose. You could drop everything and recreate it from your CREATE script, but that throws away every row you've stored. You need to change the structure in place, without losing the data inside it.

Solution: ALTER edits the definition of an existing object; DROP removes the object entirely.

We continue with the players table.

player_id username region team_id score level
1TenZNA1842Radiant
2ZekkenNA1788Radiant

ALTER TABLE — add a column

Say we now want to record each player's role.

ALTER TABLE players
ADD COLUMN role VARCHAR(20);

The column appears on every existing row with a value of NULL, because no role was known when those rows were written. Existing data is untouched.

ALTER TABLE — change a type

score was created as INT. Suppose we later need decimals.

PostgreSQL

ALTER TABLE players
ALTER COLUMN score TYPE NUMERIC(6,2);

MySQL

ALTER TABLE players
MODIFY COLUMN score DECIMAL(6,2);

This is the first real split between the two dialects you'll meet often: PostgreSQL says ALTER COLUMN ... TYPE, MySQL says MODIFY COLUMN. The database will refuse the change if the existing values can't be converted into the new type, which is the integrity promise doing its job.

ALTER TABLE — rename

ALTER TABLE players RENAME COLUMN role TO position;

ALTER TABLE players RENAME TO pro_players;

ALTER TABLE — drop a column

ALTER TABLE players DROP COLUMN position;

The column and all the values stored in it are gone. There is no per-row confirmation, so read the statement twice before running it.

DROP — removing whole objects

DROP operates on the object itself, not the rows.

DROP TABLE players;

DROP DATABASE company;

Both are destructive and final. To avoid an error when you're not sure the object exists, guard it: 

DROP TABLE IF EXISTS players;

DELETE vs TRUNCATE vs DROP

These three feel similar and are constantly confused, so it's worth lining them up. We met DELETE in the DML chapter.

Command What it removes Structure survives? Can be rolled back?
DELETE Rows (optionally some) Yes Yes, inside a transaction
TRUNCATE All rows, fast Yes Depends on the database
DROP The table itself No No

Note: Transactions using DBMS shall be discussed later.

DELETE walks the table row by row, logging each removal, which is why it is slower but also why DELETE FROM players WHERE score < 600 can target a subset and be undone. 

DELETE FROM players WHERE score < 600;

TRUNCATE empties the entire table in one stroke without logging individual rows. It is far faster, but it is all-or-nothing — there is no WHERE. 

TRUNCATE TABLE players;

DROP doesn't empty the table; it deletes the table from the database. After a DROP, even SELECT * FROM players is an error, because there is no players anymore.

Note: A useful mental model — DELETE removes some books from the shelf, TRUNCATE clears the whole shelf, DROP throws away the shelf. Reach for the gentlest one that does the job.

Summary

  • Problem: Structures change after creation, but recreating loses data.
    Solution: ALTER edits the definition in place.
  • Problem: Three different ways to "get rid of" data behave very differently.
    Solution: DELETE for some rows, TRUNCATE to empty fast, DROP to remove the object.
