Indexes

Back in the introduction, "slower search" was one of the reasons file systems lost to the DBMS — scanning records one by one to find E105. That same problem returns the moment your table is large, and the index is its answer.

Problem: SELECT * FROM players WHERE username = 'TenZ' on a table with no help has to check every row, because nothing tells the database where 'TenZ' lives. A million rows means a million comparisons — a full table scan, O(n).

Solution: An index is a separate, ordered structure that maps a column's values to the rows that hold them — like the index at the back of a book. The database searches the index instead of the table, turning O(n) into roughly O(log n).

How it works

Most indexes are B-trees: a balanced tree kept in sorted order. Because it's sorted, the database can binary-search down a handful of levels to find any value, and because it's a tree of ranges, it serves =, <, >, BETWEEN, and ORDER BY equally well. The leaf it lands on points at the actual row.

Same lookup, two strategies: the index rules out two-thirds of the tree in one hop, while a full scan has no choice but to check rows one by one.

Creating an index

MySQL / PostgreSQL

CREATE INDEX idx_username ON players(username);

Now the username lookup walks the index. A UNIQUE index does the same job while also enforcing no duplicates — which is exactly how a UNIQUE or PRIMARY KEY constraint is implemented under the hood.
Note: The primary key is always indexed; you never need to add that yourself.

B-tree is the default and the right answer almost always. Reach for a hash index only when you do nothing but exact-match lookups and have measured that it helps.

Clustered vs non-clustered

A clustered index is the table — the rows are physically stored in the index's order, so there's at most one per table. In MySQL's InnoDB, the primary key is the clustered index. A non-clustered (secondary) index is a separate structure that points back to the row. PostgreSQL stores all rows in a heap and treats every index as secondary; there is no permanent clustered index.
Knowing which engine does which explains why "look up by primary key" is often the fastest query you can write.

Composite indexes and the leftmost-prefix rule

An index can span several columns, in order:

CREATE INDEX idx_team_score ON players(team_id, score);

This index helps queries that filter on team_id, or on team_id and score — but not on score alone. A composite index is only usable from its leftmost column inward, like a phone book sorted by surname then first name: useless if you only know the first name. So column order in a composite index is a design decision, not a detail.

Covering index

If an index already contains every column a query needs, the database can answer from the index alone and never touch the table — an index-only scan. The idx_team_score index above fully answers SELECT score FROM players WHERE team_id = 1 without reading a single table row.

The cost

Indexes are not free, which is why you don't index every column:

  • Writes slow down. Every INSERT, UPDATE, or DELETE must also update every affected index. Five indexes mean five extra structures to maintain on every write.
  • Storage grows. Each index is a copy of its columns plus row pointers.
  • Low selectivity wastes effort. Indexing a column with two values (a boolean, a region with three options) barely narrows the search — the scan was nearly as fast.

The rule of thumb: index the columns you filter, join, and sort on; leave the rest alone; and let the workload, not instinct, decide.

Checking your work

Never assume an index is used — ask. EXPLAIN shows the plan the database chose: 

EXPLAIN SELECT * FROM players WHERE username = 'TenZ';

A plan that says "index scan" or "index seek" means your index is working; one that still says "seq scan" or "full table scan" means it isn't — wrong column, wrong order, or a query the index can't serve.

Summary

  • Problem: Filtering on an unindexed column scans the whole table, O(n).
    Solution: A B-tree index makes lookups, ranges, and sorts roughly O(log n).
  • Problem: Indexes cost writes and storage, and a composite index only works left-to-right.
    Solution: Index selectively, order composite columns deliberately, and verify with EXPLAIN.
