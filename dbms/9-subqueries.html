Subqueries

By now you can SELECT, filter with WHERE, and summarise with aggregates. But some questions need an answer before the main question can even run.

We'll use two tables for the rest of this page.

team_id team_name region

3 PRX APAC
4 DRX APAC

players

player_id username region team_id score level
1 TenZ NA 1 842 Radiant
2 Zekken NA 1 788 Radiant
3 Boaster EMEA 2 710 Immortal

6 something APAC 4 720 Immortal

Problem: List every player who has more score than the average score.

The obvious attempt fails:



MySQL / PostgreSQL

SELECT username, score
FROM players
WHERE score > AVG(score);   -- error

Recall the execution order: WHERE runs before aggregation. At the moment WHERE is deciding which rows to keep, AVG(score) does not exist yet. The DBMS rejects it.

Solution: Compute the average first, in its own little query, and feed the single result into the outer one. That inner query is a subquery. 



MySQL / PostgreSQL

SELECT username, score
FROM players
WHERE score > (SELECT AVG(score) FROM players);

The inner query runs first, returns 745, and the outer query then reads WHERE score > 745.

username score
TenZ 842
Zekken 788
f0rsakeN 905

A subquery is just a query wrapped in parentheses, used wherever a value, a list, or a table is expected.

Scalar subqueries

A scalar subquery returns exactly one value, so it can sit anywhere a single value can, including the SELECT list. 



MySQL / PostgreSQL

SELECT username, score,
       score - (SELECT AVG(score) FROM players) AS diff
FROM players;

Each row now carries how far it sits from the overall average. If the inner query ever returns more than one row here, the DBMS errors out, because one value was promised.

Subqueries that return a list (IN)

When the subquery returns a whole column, pair it with IN.

Problem: Show players who belong to an APAC team. The region lives on the teams table, not on players. 



MySQL / PostgreSQL

SELECT username
FROM players
WHERE team_id IN (SELECT team_id FROM teams WHERE region = 'APAC');

The inner query returns (3, 4), and the outer query keeps players whose team_id is in that list.

f0rsakeN
something
Rb

Subqueries in FROM (derived tables)

A subquery can also stand in for a whole table. This is how you filter on an aggregate that you had to compute first.

Problem: Show only regions whose average score is above 750. 



MySQL / PostgreSQL

SELECT region, avg_rr
FROM (
    SELECT region, AVG(score) AS avg_rr
    FROM players
    GROUP BY region
) AS r
WHERE avg_rr > 750;

The inner query builds a temporary table of per-region averages; the outer query treats it like any other table and filters it.

region avg_rr
NA 815

The AS r matters: a derived table must be given a name.

Correlated subqueries

So far the inner query ran once and was done. A correlated subquery is different: it references the outer row, so it re-runs once for every row the outer query looks at.

Problem: Show the highest-score player in each region. 



MySQL / PostgreSQL

SELECT username, region, score
FROM players p
WHERE score = (SELECT MAX(score) FROM players WHERE region = p.region);

For each outer player p, the inner query computes the max score of that player's region. Keep the player only if their score equals it.

username region score
TenZ NA 842
Boaster EMEA 710
Derke EMEA 710
f0rsakeN APAC 905

EMEA returns two rows because Boaster and Derke are tied. That is honest behaviour, not a bug, and it is worth remembering when you expect "the top one" and get two.

EXISTS and the NOT IN trap

EXISTS asks a yes/no question: does the inner query return at least one row? It stops at the first match, so it never bothers building a full list.

Problem: Which teams have at least one Radiant player? 



MySQL / PostgreSQL

SELECT team_name
FROM teams t
WHERE EXISTS (
    SELECT 1 FROM players
    WHERE team_id = t.team_id AND level = 'Radiant'
);
team_name
Sentinels
PRX

Now the trap. NOT IN looks like the natural opposite of IN, but it breaks the moment the inner list contains a NULL. 


SELECT username
FROM players
WHERE team_id NOT IN (SELECT team_id FROM teams WHERE region = 'EU');

If any value in that list is NULL, SQL cannot prove that a given team_id is not equal to an unknown value, so the whole condition collapses to "unknown" and the query returns zero rows. This is three-valued logic: a comparison with NULL is neither true nor false. The safe habit is NOT EXISTS, which is immune to it: 


SELECT username
FROM players p
WHERE NOT EXISTS (
    SELECT 1 FROM teams WHERE team_id = p.team_id AND region = 'EU'
);

Optimisation

Many subqueries can be rewritten as JOINs, and the planner often does exactly that behind the scenes. But not always, and a correlated subquery is the one to watch: it re-runs for every outer row, so a 10,000-row table can mean 10,000 inner executions. When a query feels slow, run EXPLAIN in front of it and read whether the database chose a per-row loop or a single pass. Prefer EXISTS / NOT EXISTS over IN / NOT IN once NULLs are in play, both for correctness and because the engine can short-circuit on the first match.

Summary

  • Problem: A query needs a value that can only be known after another query runs.
    Solution: Nest that query as a subquery; the inner one runs first.
  • Problem: The opposite of IN silently returns nothing.
    Solution: NULLs poison NOT IN; use NOT EXISTS.
