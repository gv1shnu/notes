Constraints and Actions

Difficulty: Medium

In the introduction we listed integrity problems as one of the reasons file-based systems gave way to the DBMS. Constraints are how that promise is kept. We've already used one of them without naming it as a family member: the datatype.

Problem: Left to itself, the database will faithfully store nonsense — a negative score, two players sharing one id, a team_id pointing at a team that doesn't exist, a row with no username at all. The datatype stops "hello" going into an integer column, but it can't stop any of those.

Solution: Constraints are rules attached to the table that the DBMS enforces automatically, on every insert and update, forever. You declare the rule once; you never have to remember to check it again.

We'll build the players and teams tables properly this time.

The constraints

Constraint Rule it enforces
PRIMARY KEYUniquely identifies a row (unique + not null)
NOT NULLA value must be present
UNIQUENo two rows share this value
CHECKA value must satisfy a condition
DEFAULTSupplies a value when none is given
FOREIGN KEYA value must reference a real row elsewhere

Here they are at work.

CREATE TABLE teams (
    team_id   INT PRIMARY KEY,
    team_name VARCHAR(30) NOT NULL UNIQUE,
    region    VARCHAR(10) NOT NULL
);

CREATE TABLE players (
    player_id INT PRIMARY KEY,
    username  VARCHAR(30) NOT NULL UNIQUE,
    score        INT DEFAULT 0 CHECK (score >= 0),
    level      VARCHAR(20) DEFAULT 'Unranked',
    team_id   INT REFERENCES teams(team_id)
);

Reading that second table top to bottom: every player must have a unique id and a unique, present username; score can't be negative and starts at 0 if you don't supply it; level defaults to 'Unranked'; and team_id, if given, must match a real team.

A few now fail by design:

INSERT INTO players (player_id, username, score) VALUES (1, 'TenZ', -50);
-- rejected: CHECK (score >= 0)

INSERT INTO players (player_id, username) VALUES (2, 'TenZ');
-- rejected: UNIQUE on username

INSERT INTO players (player_id, username, team_id) VALUES (3, 'Derke', 99);
-- rejected: no team 99 exists

The foreign key, properly

The FOREIGN KEY is the constraint that makes a relational database "relational". It says a column's values must already exist as keys in another table. players.team_id references teams.team_id, so a player can only belong to a team that has actually been created. This is referential integrity — no orphans, no dangling pointers.

But it raises a question the other constraints don't. A player depends on a team. So what happens to the players when that team is deleted, or when its team_id changes?

Actions — ON DELETE and ON UPDATE

This is the part most people skip and later regret. When you declare a foreign key, you also decide what the database should do to the child rows when the parent row is deleted or updated.

Action When the parent row is deleted/updated...
RESTRICTRefuse the operation while children still reference it
NO ACTIONSame effect as RESTRICT (the default if unspecified)
CASCADEDelete or update the children to match
SET NULLSet the children's foreign key to NULL
SET DEFAULTSet the children's foreign key to its default value

You attach the action when defining the key:

CREATE TABLE players (
    player_id INT PRIMARY KEY,
    username  VARCHAR(30) NOT NULL UNIQUE,
    team_id   INT REFERENCES teams(team_id)
              ON DELETE SET NULL
              ON UPDATE CASCADE
);

With this, deleting Sentinels doesn't delete TenZ — his team_id simply becomes NULL, a free agent. And if Sentinels' team_id is renumbered, TenZ's row follows it automatically.

Pick the action by asking what the data means. If a player is meaningless without a team, ON DELETE CASCADE is honest. If a player outlives their team, SET NULL is honest. If deleting a team that still has players is a mistake you want to catch, RESTRICT is honest. The default, NO ACTION, quietly blocks the delete — which surprises people who never chose it.

Adding constraints later

Constraints aren't locked in at creation. Using ALTER from the previous chapter, you can add or drop them as the schema matures, and it's good practice to name them so the error messages mean something. 

ALTER TABLE players
ADD CONSTRAINT chk_rr CHECK (score BETWEEN 0 AND 1000);

ALTER TABLE players
DROP CONSTRAINT chk_rr;

Summary

  • Problem: The database will store invalid and orphaned data unless told the rules.
    Solution: Constraints enforce those rules automatically, on every write.
  • Problem: Deleting a parent row leaves its children in an undefined state.
    Solution: Referential actions (CASCADE, SET NULL, RESTRICT) decide what happens, and the right one depends on what the data means.
