Before we move to dealing with multiple tables, we need to understand the concept of keys, which was briefly mentioned before.

Keys

Difficulty: Medium

Problem: A table is just rows of data, and data repeats. Two players can share a score, a region, even a name. So how do you point at exactly one row — the row you want to update, delete, or link to from another table — and be sure you'll never hit the wrong one?

Solution: A key is one or more columns whose value is guaranteed to identify a row uniquely. That guarantee is what UPDATE, DELETE, and especially joins rely on. The seven "types" of keys below are not seven different things — they are seven roles the same idea plays.

  1. Super Key
  2. Candidate Key
  3. Primary Key
  4. Unique Key
  5. Composite Key
  6. Foreign Key
  7. Logical Key

We'll use this players table throughout.

player_id username region team_id score
1TenZNA1842
2ZekkenNA1788
3BoasterEMEA2710

1. Super Key

A super key is any set of columns that identifies a row uniquely. "Any set" is the operative phrase — it is allowed to carry extra, useless columns, as long as the whole set stays unique.

Since player_id is unique on its own, every one of these is a super key:

  • { player_id }
  • { player_id, score }
  • { player_id, username, region }
  • { username } (usernames are unique here too)

The middle two work only because player_id is already unique; score and region are dead weight. That dead weight is exactly what the next key removes.

2. Candidate Key

A candidate key is a super key with nothing to spare — a minimal one. Remove any column and it stops being unique. A table can have several.

In players, both { player_id } and { username } are candidate keys: each is unique, and neither has a column you could drop. { player_id, score } is not a candidate key — it's a super key, but player_id alone already does the job, so score is spare.

Candidate keys are the shortlist. From them, you pick one to be the row's official identity.

3. Primary Key

The primary key is the candidate key you elect as the identifier for the table. There is exactly one, it cannot be NULL, and no two rows may share it.

MySQL / PostgreSQL

CREATE TABLE players (
    player_id INT PRIMARY KEY,
    username  VARCHAR(30) NOT NULL UNIQUE
);

The candidate keys you didn't choose don't disappear — they become alternate keys. Here username is an alternate key, still unique, just not the chosen identity. (That's the eighth term people often count separately; it's really just "a candidate key that lost the election.")

4. Unique Key

A unique key enforces "no duplicates" on a column without making it the primary key. The difference matters:

  • A table has one primary key but can have many unique keys.
  • A primary key forbids NULL; a unique key allows it — and in most databases, more than one NULL, since NULL is never equal to NULL.

In the table above, username is declared UNIQUE — every player's name stays distinct, but it isn't carrying the table's identity. Primary key = unique + not null + chosen. Unique key = just unique.

5. Composite Key

Sometimes no single column is unique, but a combination is. A key built from two or more columns is a composite key.

Consider a table recording each player's stats per match:

match_idplayer_idkills
M1124
M1219
M2130

match_id repeats (one match, many players) and player_id repeats (one player, many matches). Neither alone identifies a row. Together, (match_id, player_id) does — one player appears once per match.

MySQL / PostgreSQL

CREATE TABLE match_stats (
    match_id  INT,
    player_id INT,
    kills     INT,
    PRIMARY KEY (match_id, player_id)
);

6. Foreign Key

Everything so far identifies a row within one table. A foreign key reaches across tables: it's a column whose values must match a key in another table.

players.team_id is a foreign key referencing teams.team_id. A player can only sit on a team that exists — no orphans. This is the bridge between the two tables, and it's the reason the next chapter on joins is even possible.

MySQL / PostgreSQL

team_id INT REFERENCES teams(team_id)

We met foreign keys as a constraint earlier; here the point is narrower — a foreign key is how one table points at another's primary key.

7. Logical Key

Two players are never the same person, even if a botched import happens to hand them the same auto-generated player_id. A logical key (also called a natural or business key) is an identifier that comes from the real world and carries meaning — a Riot ID, an email, a username.

Contrast it with a surrogate key: an artificial number the database invents purely to have a stable handle, with no meaning of its own. 

player_id SERIAL PRIMARY KEY   -- surrogate: meaningless, but stable

The trade-off: a natural key is meaningful but can change — a player renames themselves and every reference to the old name breaks — while a surrogate key never changes but tells you nothing. A common design uses a surrogate as the primary key and keeps the logical key as a UNIQUE column: stability and meaning, both.

Summary

Key What it is
Super KeyAny column set that identifies a row uniquely (extras allowed)
Candidate KeyA super key with nothing to spare (minimal)
Primary KeyThe candidate key chosen as identity (unique + not null, one per table)
Unique KeyEnforces uniqueness, allows NULLs, many per table
Composite KeyA key built from two or more columns together
Foreign KeyA column that must match a key in another table
Logical KeyA real-world identifier (natural), as opposed to a surrogate id

With keys understood — and foreign keys in particular — we can finally connect tables to each other.
