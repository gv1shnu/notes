Concurrency Control

The Isolation in ACID promised that concurrent transactions wouldn't see each other's half-done work. This is the machinery that keeps that promise — and the dials it gives you.

Problem: Two transactions touch the same row at the same time. TenZ has 800 points; transaction A adds a 50-point bonus, transaction B adds a 30-point bonus. Both read 800, both compute from it, both write back. One write lands on top of the other, and 30 (or 50) points silently disappear. This is the lost update, and it's only the gentlest of the things that go wrong when transactions interleave unmanaged.

Solution: Concurrency control schedules concurrent transactions so the final result is the same as if they had run one after another — a serializable outcome. Two families of technique deliver it: locking and multi-version concurrency control.

What can go wrong: the read phenomena

The SQL standard names three ways one transaction can be polluted by another:

  • Dirty read: A reads a row that B has changed but not committed. B rolls back, and A acted on data that never officially existed.
  • Non-repeatable read: A reads a row, B updates and commits it, A reads the same row again and gets a different value — within one transaction.
  • Phantom read: A runs WHERE score > 800, B inserts a new qualifying row and commits, A re-runs the query and a new "phantom" row has appeared.

Isolation levels

You choose how much protection you want — and pay for it in concurrency. Each level forbids more phenomena than the last.

The defaults differ and catch people out: PostgreSQL defaults to READ COMMITTED, while MySQL's InnoDB defaults to REPEATABLE READ (and, unusually, prevents phantoms at that level too). Know your engine's default before you reason about a bug.

Locking

The classic mechanism. A transaction takes a shared (S) lock to read and an exclusive (X) lock to write. Many transactions can share an S lock on the same row, but an X lock blocks everyone else — readers and writers alike.

To guarantee serializability, databases follow two-phase locking (2PL): a transaction first only acquires locks (the growing phase), then only releases them (the shrinking phase), never reacquiring once it has let go. Strict 2PL — holding every lock until commit — is what most engines actually use.

You can also lock explicitly when you know a read-modify-write is coming:

SELECT score FROM players WHERE username = 'TenZ' FOR UPDATE;

FOR UPDATE takes the exclusive lock now, so transaction B in our lost-update story has to wait its turn instead of overwriting A.

Deadlocks

Locking has a price. Transaction A holds a lock B wants, and B holds a lock A wants — neither can proceed, forever. That's a deadlock. Databases resolve it by detecting the cycle (a wait-for graph) or by timing out, then aborting one transaction as the victim and rolling it back so the other can finish. You avoid them by acquiring locks in a consistent order everywhere in your code.

MVCC

Locking makes readers and writers block each other, which hurts. Multi-Version Concurrency Control avoids most of that: instead of overwriting a row, the database keeps multiple versions of it, and each transaction reads a consistent snapshot as of the moment it started. Readers never block writers and writers never block readers — they just see different versions.

This is why, back in the transactions chapter, a PostgreSQL UPDATE was described as writing a new row version rather than editing in place: that old version is still there for any transaction that began before the change. PostgreSQL and InnoDB both lean on MVCC.

The trade-off

Stronger isolation buys correctness with throughput. SERIALIZABLE is the safest and the slowest, because it must serialize conflicting work; READ COMMITTED lets far more run in parallel but admits non-repeatable and phantom reads. The right level is the weakest one your operation can tolerate — not the strongest you can name.

Summary

  • Problem: Interleaved transactions corrupt each other — lost updates, dirty, non-repeatable, and phantom reads.
    Solution: Concurrency control schedules them to a serializable outcome via locking or MVCC.
  • Problem: How much isolation is enough?
    Solution: Pick an isolation level — the weakest that still keeps your operation correct.
