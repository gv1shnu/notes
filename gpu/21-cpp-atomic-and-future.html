Parallel syntax — C++

21. atomic & future.

Atomic variables fix the lost-increment problem: a variable cannot be touched until the current operation on it finishes. Incrementing is really read + add + write — without atomicity, two increments can collapse into one.

Tool Effect
std::atomic read-modify-write happens as one indivisible step
atomic_thread_fence acquire / release barrier so threads reach a point before proceeding

Don't make every variable atomic — only the ones two threads truly share.

Futures let a thread keep working while a result is still pending (think: a network call).

Tool Effect
std::async run a function outside the caller; returns a future
future.wait() block until the result is ready
future.get() retrieve the result when it arrives
In code 
C++

#include <atomic>
#include <future>

std::atomic<int> count{0};

int compute() { return 42; }

int main() {
    count++;                          // atomic, safe
    std::future<int> f = std::async(compute);
    int result = f.get();             // wait + retrieve
}
