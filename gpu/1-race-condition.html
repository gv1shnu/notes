Pitfalls of shared state

Difficulty: Medium

Threads run free until they touch the same data. Then order of execution decides the result.

1. Race condition.

Two threads update a shared variable; their instructions interleave, so the final value is not the one intended.

Shared count = 0. Each thread does count = count + 1. Expected after both: 2.

Both threads read 0 before either writes, so each computes 0+1 and stores 1. B's write lands on top of A's — one increment is silently lost. The fix is to make read-modify-write atomic (a lock or an atomic add) so the second thread can't read until the first has written.

Step Thread A Thread B count
1 read count (0) 0
2 read count (0) 0
3 add 1 → 1 0
4 write 1 1
5 add 1 → 1 1
6 write 1 1

One update was lost. Note: neither thread is "wrong" — each read a valid value. The order was the bug.


Solution
Mechanism Effect
Lock / Mutex Only one thread holds count at a time
Atomic operation read → add → write becomes indivisible

Minimize shared variables. If shared, make each thread's access atomic.
