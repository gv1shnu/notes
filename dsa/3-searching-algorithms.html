Searching Algorithms

Find a key in a collection. The whole story is whether the data is sorted — that one fact is the difference between O(n) and O(log n).

Linear search — O(n)

Walk the array front to back; return the index on a match, else -1. Works on any array (sorted or not) and on linked lists. Worst case (key absent) checks all n elements. 

def linear_search(A, key):
    for i in range(len(A)):   # walk front to back
        if A[i] == key:
            return i             # found
    return -1                    # absent

int linear_search(const vector<int>& A, int key) {
    for (int i = 0; i < (int)A.size(); ++i)  // walk front to back
        if (A[i] == key) return i;              // found
    return -1;                                  // absent
}

Recurrence: T(n) = T(n-1) + O(1) = O(n) — technically a (degenerate) divide-and-conquer: the subproblem shrinks by just one each step.

Binary search — O(log n)

On a sorted array, halve the search space each step: compare the key to the midpoint and discard the half it can't be in. 

def binary_search(A, key):       # A must be sorted
    lo, hi = 0, len(A) - 1
    while lo <= hi:
        mid = (lo + hi) // 2       # no overflow in Python
        if A[mid] == key:
            return mid
        elif key < A[mid]:
            hi = mid - 1
        else:
            lo = mid + 1
    return lo                     # insertion point if absent

int binary_search_idx(const vector<int>& A, int key) {  // A sorted
    int lo = 0, hi = (int)A.size() - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;        // avoids int overflow
        if (A[mid] == key) return mid;
        else if (key < A[mid]) hi = mid - 1;
        else                  lo = mid + 1;
    }
    return lo;                          // insertion point if absent
}

Recurrence: T(n) = T(n/2) + O(1) = O(log n). The iterative form (above) avoids the recursion stack. When absent, lo is the insertion point — where the key would go to keep the array sorted — so callers learn more than a bare "not found". (Note the C++ overflow-safe midpoint lo + (hi - lo) / 2; Python's unbounded ints make (lo + hi) // 2 safe.)

Search Needs sorted? Time
Linear no O(n)
Binary yes O(log n)

Why it matters: a 50,000-word dictionary lookup drops from ~50,000 comparisons to ~16 — "ten seconds" to "instant". The catch is keeping data sorted; if sorting breaks a needed correspondence, store a sorted array of indices (a space-for-time trade) instead.

Language notes

You rarely hand-roll these — both languages ship binary search. Reach for the library, but know what's underneath:

Call Time Under the hood
bisect.bisect_left / bisect_right(a, x) O(log n) binary search on a sorted list; returns the insertion index (C-accelerated)
bisect.insort(a, x) O(n) finds the spot in O(log n), but the list insert shifts elements → O(n)
x in a (on a list) O(n) linear scan — for membership use a set/dict (O(1) average)
Call Time Under the hood
std::lower_bound / upper_bound O(log n) first/last slot for a key; needs a sorted random-access range (degrades on a list)
std::binary_search O(log n) returns only bool; on a std::set/map use the container's own .find()
std::find O(n) linear scan over any range — the std equivalent of linear search

Either way: the library binary search assumes the range is already sorted and randomly indexable — the moment you hand it a linked structure, the O(log n) promise quietly becomes O(n).
