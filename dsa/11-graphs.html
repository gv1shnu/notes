Graphs

Difficulty: Medium

A set V of vertices plus a set E of edges, each joining a pair of vertices. Models anything that's "connections between things" — maps (intersections + roads), the web (pages + links), social networks (people + friendships). Undirected edges go both ways; directed ones have an orientation. A simple graph has no self-loops and no parallel edges.

Problem: how do you even store one?

Before any algorithm, you need the graph in memory — and the operation you do most decides the layout. Three options, and they're not interchangeable:

Operation Edge list Adjacency matrix Adjacency list
Is u–v an edge? O(|E|) O(1) O(deg v)
List all neighbors of v O(|E|) O(|V|) O(deg v)
List all edges O(|E|) O(|V|²) O(|E|)
Space O(|E|) O(|V|²) O(|V| + |E|)

Almost every algorithm repeatedly asks "give me the neighbors of v," and real graphs (maps, web, social) are sparse (|E| ≈ |V|, not |V|²). Both facts point the same way: use the adjacency list. The matrix only earns its O(1) edge test on dense graphs.

Problem: what can I reach from here?

Walk as far forward as you can, back up one step at a dead end, and never revisit a marked vertex — a depth-first walk. The recursion rides the call stack, so the frontier is implicit: 

graph = {0: [1, 2], 1: [0], 2: [0]}   # adjacency list: dict -> list

def explore(v, visited, comp, label):  # DFS, marks reachable from v
    visited.add(v); comp[v] = label
    for w in graph[v]:
        if w not in visited:
            explore(w, visited, comp, label)

def connected_components(graph):
    visited, comp, label = set(), {}, 0
    for v in graph:
        if v not in visited:
            explore(v, visited, comp, label)
            label += 1                # bump before the next island
    return comp

vector<vector<int>> graph;            // graph[v] = neighbors of v

void explore(int v, vector<int>& comp, int label) {  // DFS
    comp[v] = label;                  // comp[v] == -1 means unvisited
    for (int w : graph[v])
        if (comp[w] == -1) explore(w, comp, label);
}
vector<int> connected_components() {
    int n = graph.size();
    vector<int> comp(n, -1);
    int label = 0;
    for (int v = 0; v < n; ++v)
        if (comp[v] == -1) { explore(v, comp, label); ++label; }
    return comp;
}

explore(s) marks exactly the vertices reachable from s. Wrap it — for every vertex, if unvisited, explore it — and you get depth-first search (DFS), which sweeps the whole graph (even disconnected pieces). Each vertex is explored once and each edge inspected twice: O(|V| + |E|), linear.

Problem: which pieces are disconnected?

In an undirected graph "reachable from" is an equivalence relation, so the graph splits into connected components — islands where any two vertices reach each other and no edge crosses between. You get them from DFS for free: keep a counter, stamp every vertex an explore finds, and bump the counter before each fresh explore. Same O(|V| + |E|). Capturing the search order: pre/post numbers

DFS visits vertices in a meaningful order, then throws it away. Capture it: run a clock that ticks on entry to and exit from each explore, recording pre(v) and post(v). Every vertex now owns an interval [pre, post] — and these intervals have one rigid property that later algorithms (topological sort, SCCs) lean on:

Two intervals are always nested (one explore ran as a subroutine of the other) or disjoint (neither was open when the other was found) — never half-overlapping. That "nested-or-disjoint" guarantee is exactly what makes pre/post numbers a trustworthy read on the search's shape.

Language notes

  • Store the graph as dict[node] -> list (or collections.defaultdict(list)); nodes can be any hashable label, not just 0..n-1.
  • Recursive DFS hits the ~1000-frame recursion limit on deep/long graphs — convert to an explicit list stack, or raise sys.setrecursionlimit.
  • Weighted edges: store (neighbor, weight) tuples in the adjacency lists.
  • vector<vector<int>> for vertices 0..n-1; vector<vector<pair<int,int>>> when edges carry weights.
  • For sparse or arbitrary labels, unordered_map<int, vector<int>>.
  • Recursive DFS can blow the call stack on huge graphs — an explicit std::stack is the safe default at scale.
