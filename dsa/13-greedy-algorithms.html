Greedy Algorithms

Make the choice that looks best right now, shrink the problem, repeat. It's fast and simple — but only correct when the greedy choice is provably safe.

The recipe

Step What
1. Greedy choice commit to a first choice that looks best
2. Prove it's safe show some optimal solution is consistent with that choice
3. Reduce the choice leaves a smaller subproblem of the same kind
4. Repeat solve the subproblem the same way, until nothing's left

Safe is the whole game: a greedy choice is safe if there's an optimal solution that includes it. Many tempting greedy choices are not safe — you must prove it before trusting it.

Where greedy works: fractional knapsack

A burglar with a 15 kg sack and divisible goods (spices) wants maximum value. Safe choice: take as much as possible of the item with the highest value per kg, then the next, and so on.

Item Value / weight Take order
spice C $8/kg 1st (all)
spice B $7/kg 2nd (all)
spice A $6/kg 3rd (fill the rest)

Proof sketch: if an optimal solution skips some of the best-per-kg item, swap an equal weight of a worse item for it — value goes up (or stays), so it was never worse to be greedy. (Note: this is the fractional knapsack. The 0/1 version — whole items only — is not greedy-safe and needs dynamic programming.)

Where greedy beats brute force

The celebration party problem: split children into the fewest groups so any two in a group differ by ≤ 2 years. Brute force tries every split — Ω(2n), hopeless past ~50 kids. Greedy (sort by age, start a group at the youngest ungrouped child, sweep up everyone within 2 years, repeat) is near-linear and provably optimal.

Watch out: greedy isn't universal. The coin-change problem is greedy-safe with US coins but fails in Tanzania (a 20¢ coin makes 40¢ = 20+20 beat the greedy 25+10+5). When greedy isn't safe, reach for dynamic programming.
