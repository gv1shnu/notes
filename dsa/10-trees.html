Trees

Difficulty: Medium

A root with children, each a subtree. A binary tree has ≤ 2 children per node. In CS, trees grow downward.

Traversals

Traversal Order Uses
Pre-order (DFS) node, left, right copy a tree
In-order (DFS) left, node, right BST → sorted output
Post-order (DFS) left, right, node evaluate / delete bottom-up
Level-order (BFS) level by level shortest paths

Depth-first is naturally recursive (it rides the call stack); breadth-first uses an explicit queue: enqueue the root, then repeatedly dequeue a node and enqueue its children. 

def inorder(node, visit):          # left, node, right -> sorted for a BST
    if node is None: return
    inorder(node.left, visit)
    visit(node.key)
    inorder(node.right, visit)

def level_order(root, visit):      # BFS, uses a queue
    from collections import deque
    q = deque([root] if root else [])
    while q:
        n = q.popleft(); visit(n.key)
        if n.left:  q.append(n.left)
        if n.right: q.append(n.right)

void inorder(Node* node, vector<int>& out) {  // left, node, right
    if (!node) return;
    inorder(node->left, out);
    out.push_back(node->key);
    inorder(node->right, out);
}
void level_order(Node* root, vector<int>& out) {   // BFS
    queue<Node*> q; if (root) q.push(root);
    while (!q.empty()) {
        Node* n = q.front(); q.pop(); out.push_back(n->key);
        if (n->left)  q.push(n->left);
        if (n->right) q.push(n->right);
    }
}

Watch it run

Keys inserted into a BST one at a time — each walks down (< goes left, > goes right) and attaches where it falls off. Then an in-order traversal (left, node, right) reads the keys back out, and they come out sorted — that's the BST invariant paying off. Reset shuffles the insert order (different shape, same sorted output).

Binary search trees

A BST keeps everything in the left subtree ≤ the node ≤ everything in the right. So Find, Insert, and Delete just walk down comparing — cost = O(depth). (In-order traversal yields sorted order, by definition.)

The catch: depth can be terrible. Insert 1,2,3,4,5 in order and the tree degenerates into a chain → O(n). We need it balanced (each node's two subtrees ~equal size) for O(log n). 

def find(node, key):               # O(height)
    while node and node.key != key:
        node = node.left if key < node.key else node.right
    return node

def insert(root, key):             # O(height)
    if root is None: return Node(key)
    if   key < root.key: root.left  = insert(root.left, key)
    elif key > root.key: root.right = insert(root.right, key)
    return root                     # equal key: ignore (no dups)

Node* find(Node* node, int key) {  // O(height)
    while (node && node->key != key)
        node = key < node->key ? node->left : node->right;
    return node;
}
Node* insert(Node* root, int key) {  // O(height)
    if (!root) return new Node(key);
    if      (key < root->key) root->left  = insert(root->left, key);
    else if (key > root->key) root->right = insert(root->right, key);
    return root;                    // equal key: ignore (no dups)
}

Rotations

A rotation locally rearranges a parent/child pair (and re-hangs one subtree) to change the shape while preserving the sort order. It's O(1), and it's the tool every self-balancing tree uses to fix imbalance.

AVL trees

A BST that stores each node's height and maintains the AVL property: at every node, |height(left) - height(right)| ≤ 1. That bound forces height = O(log n) (proof: an AVL tree of height h has at least Fh nodes — Fibonacci, which grows exponentially). After each insert/delete, walk back up and rotate to restore the property.

Operation Time
Find / Insert / Delete O(log n)

Splay trees

Self-adjusting: every time you access a node, splay it to the root via rotations (the zig, zig-zig, and zig-zag cases). Frequently-used items drift near the top, so they're cheap to reach — O(log n) amortized, often faster on skewed access patterns. (Naively rotating straight to the root doesn't work — it can stay unbalanced and hit O(n); the zig-zig/zig-zag cases are what make it work.)

Language notes

  • No built-in balanced BST. For sorted/ordered needs reach for the third-party sortedcontainers (SortedList, SortedDict) — O(log n) add/remove, backed by a list of lists, not a tree.
  • Unordered lookups go to dict/set (hashing, O(1) average); priority needs go to heapq.
  • Recursive traversals risk RecursionError on a skewed (chain-like) tree — iterate with an explicit stack for deep inputs.
  • std::set / map (and the multi variants) are balanced BSTs (red-black trees): O(log n) insert/find/erase, iteration in sorted order.
  • lower_bound / upper_bound on them give O(log n) range queries — the payoff over a hash map.
  • Tree nodes are scattered in memory, so for pure lookup a unordered_map usually wins; use the tree when order matters.
