Sorting Algorithms

Put a sequence in order. The quadratic ones are simple; the O(n log n) ones win at scale — and n log n is as good as comparison sorting can get.

Selection sort — O(n²)

Grow a sorted prefix: find the minimum of the unsorted part, swap it to the front, repeat. 

def selection_sort(A):
    n = len(A)
    for i in range(n):
        m = i                      # index of min in A[i:]
        for j in range(i + 1, n):
            if A[j] < A[m]: m = j
        A[i], A[m] = A[m], A[i]    # swap min to front

void selection_sort(vector<int>& A) {
    int n = A.size();
    for (int i = 0; i < n; ++i) {
        int m = i;                 // index of min in A[i..]
        for (int j = i + 1; j < n; ++j)
            if (A[j] < A[m]) m = j;
        swap(A[i], A[m]);          // swap min to front
    }
}

Two nested loops → (n-1)+(n-2)+…+1 = n(n+1)/2 = O(n²), regardless of input. Sorts in place (no extra memory). Insertion sort and bubble sort are the same O(n²) class.

Merge sort — O(n log n)

Divide and conquer: split in half, sort each half recursively, then merge the two sorted halves in one linear pass. 

def merge_sort(A):
    if len(A) <= 1: return A
    mid = len(A) // 2
    L, R = merge_sort(A[:mid]), merge_sort(A[mid:])
    return merge(L, R)

def merge(L, R):                   # one linear pass
    out, i, j = [], 0, 0
    while i < len(L) and j < len(R):
        if L[i] <= R[j]: out.append(L[i]); i += 1
        else:            out.append(R[j]); j += 1
    out.extend(L[i:]); out.extend(R[j:])
    return out

void merge_sort(vector<int>& A, int lo, int hi) {  // [lo, hi)
    if (hi - lo <= 1) return;
    int mid = lo + (hi - lo) / 2;
    merge_sort(A, lo, mid);
    merge_sort(A, mid, hi);
    vector<int> tmp; tmp.reserve(hi - lo);
    int i = lo, j = mid;
    while (i < mid && j < hi)            // one linear pass
        tmp.push_back(A[i] <= A[j] ? A[i++] : A[j++]);
    while (i < mid) tmp.push_back(A[i++]);
    while (j < hi)  tmp.push_back(A[j++]);
    for (int k = 0; k < (int)tmp.size(); ++k) A[lo + k] = tmp[k];
}

T(n) = 2T(n/2) + O(n) = O(n log n) — log n levels, O(n) work each. Fast enough to sort a million items on a laptop (where O(n²) would need ~1012 ops). Needs O(n) extra space for the merge. (Full walk-through under Divide & Conquer.)

Quicksort — O(n log n) average

Pick a pivot, partition the array so smaller elements go left and larger go right (pivot lands in its final spot), then recurse on each side. Sorts in place, no merge. 

def quicksort(A, lo, hi):          # inclusive bounds
    if lo >= hi: return
    p = partition(A, lo, hi)
    quicksort(A, lo, p - 1)
    quicksort(A, p + 1, hi)

def partition(A, lo, hi):
    pivot = A[hi]                  # last element as pivot
    i = lo
    for j in range(lo, hi):
        if A[j] < pivot:
            A[i], A[j] = A[j], A[i]; i += 1
    A[i], A[hi] = A[hi], A[i]      # pivot to its spot
    return i

int partition(vector<int>& A, int lo, int hi) {
    int pivot = A[hi], i = lo;     // last element as pivot
    for (int j = lo; j < hi; ++j)
        if (A[j] < pivot) swap(A[i++], A[j]);
    swap(A[i], A[hi]);             // pivot to its spot
    return i;
}
void quicksort(vector<int>& A, int lo, int hi) {  // inclusive
    if (lo >= hi) return;
    int p = partition(A, lo, hi);
    quicksort(A, lo, p - 1);
    quicksort(A, p + 1, hi);
}
Case Time
average (random pivot) O(n log n)
worst (already sorted, bad pivot) O(n²)

A random pivot (shuffle, or pick a random index to swap in) makes the worst case practically impossible; in practice quicksort is the fastest general sort (cache-friendly, in place).

The n log n wall

Any comparison-based sort must take Ω(n log n) in the worst case. Picture its decision tree: each comparison is a node, each leaf an output ordering. There are n! possible orderings, so ≥ n! leaves; a binary tree with n! leaves has depth ≥ log(n!) = Ω(n log n). So merge sort is asymptotically optimal — you can't beat it by comparing.

Beating the wall: counting sort — O(n + M)

If you don't compare — possible when keys are small integers in 0..M — you can go linear. Count occurrences of each value in one pass, then write them out in order. 

def counting_sort(A, M):           # keys in 0..M
    count = [0] * (M + 1)
    for x in A: count[x] += 1      # one pass
    out = []
    for v in range(M + 1):
        out.extend([v] * count[v]) # write in order
    return out

vector<int> counting_sort(const vector<int>& A, int M) {  // keys 0..M
    vector<int> count(M + 1, 0);
    for (int x : A) count[x]++;   // one pass
    vector<int> out;
    for (int v = 0; v <= M; ++v)
        for (int c = 0; c < count[v]; ++c) out.push_back(v);  // in order
    return out;
}

O(n + M); linear when M = O(n). Radix sort extends this to larger keys digit by digit. The lower bound doesn't apply — no comparisons were made.

Language notes

  • sorted(it) / list.sort() use Timsort: stable, adaptive (O(n) on nearly-sorted data), O(n log n) worst, O(n) extra space.
  • key= computes each element's sort key once (decorate–sort–undecorate); reverse=True for descending.
  • list.sort() mutates in place and returns None; sorted() returns a new list and works on any iterable.
  • std::sort = introsort (quicksort, heapsort fallback, insertion for small ranges): O(n log n) worst, in place, but not stable.
  • std::stable_sort keeps equal elements' order (merge-based, O(n) extra memory); std::sort(begin, end, cmp) takes a comparator.
  • For partial work: std::partial_sort (top k) and std::nth_element (O(n) selection) beat a full sort.
