Introduction

The weight of this subject has changed from computer science core to MAANG core in the past decade. Literally nobody learns this subject just for the sake of learning or experimenting. The only reason to learn this subject is to get a job, and corporates don't seem to be avoiding this truth either, pushing more and more learners towards this subject, with or without their genuine interest. So now, the reason to learn this subject is more of a survival decision for CS grads.

The good news is, this subject is not as hard as it seems. It is just a matter of understanding the concepts and practicing them. The bad news is, it is not as easy as it seems. It requires a lot of practice and patience to master this subject.

The whole subject revolves around how to manipulate and fetch data in the least amount of time using optimal amount of space. So, the main focus of this subject is to understand the trade-offs between time and space complexity, and how to optimize them. So, we shall start with Time and Space complexity, and then move on to the different data structures and algorithms, and how to analyze them.

Time and Space Complexity

The same problem can have two correct algorithms a lifetime apart. Take Fibonacci. The naive recursion is just the definition: fib(n) = fib(n-1) + fib(n-2). It's correct — and useless. It rebuilds the same sub-results over and over (a giant tree of repeated calls), so its work grows like the Fibonacci numbers themselves.

Compute F(100) Cost
naive recursion ~1.77 × 1021 steps → ~56,000 years at 1 GHz
fill an array left→right ~200 steps → instant

So we don't measure wall-clock seconds — those depend on the machine, compiler, and architecture, which only change the answer by a constant. We measure how runtime scales with input size, and ignore constants. That's Big-O:

f(n) = O(g(n))  means  f(n) ≤ c · g(n) for some constant c and all large n. A few rules cover most cases:

Rule Example
drop constant factors 7n³ = O(n³)
drop lower-order terms 3n² + 5n + 2 = O(n²)
any exponential beats any polynomial n100 = O(1.1n)
any polynomial beats any power of log (log n)³ = O(√n)

Why the growth rate is everything — what you can finish in one second at ~1 GHz:

Runtime Max input n
n ~1,000,000,000
n log n ~30,000,000
n² ~30,000
2n ~30

Growth order, best to worst: 1 < log n < √n < n < n log n < n² < 2n < n!. Related notation: Ω is a lower bound (grows no slower), Θ is tight (same rate both ways), little-o is strictly slower. Big-O is the one we'll use for almost everything — just remember it hides constants, so the last factor-of-2 still needs real measurement.

With that, the three great algorithm-design paradigms: Greedy, Divide & Conquer, and Dynamic Programming.
