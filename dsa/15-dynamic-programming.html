Dynamic Programming

When a problem breaks into overlapping subproblems, plain recursion recomputes the same answers a trillion times. DP solves each subproblem once, stores it in a table, and builds up — usually bottom-up, smallest first.

(The name is a smokescreen: Richard Bellman picked "dynamic programming" so his maths would sound impressive to a defense budget committee. It has nothing to do with programming.)

Change problem — the table idea

Fewest coins to make m from given denominations. Greedy fails for some coin systems; recursion is correct but exponential (it re-solves the same amounts endlessly). DP fills a table left to right: 

minCoins[0] = 0
for amount = 1 .. m:
    minCoins[amount] = 1 + min( minCoins[amount - c] for each coin c ≤ amount )

Each amount is solved once by looking up smaller amounts already computed. O(m · #coins).

Edit distance — a 2D grid

Fewest insert/delete/substitute edits to turn string A into string B. Let D(i,j) = edit distance of the first i of A and first j of B. Each cell looks at three neighbours: 

D(i,j) = min(
    D(i-1, j)   + 1,                 # deletion
    D(i,   j-1) + 1,                 # insertion
    D(i-1, j-1) + (A[i] != B[j])     # match (0) or substitution (1)
)
Base Value
D(i, 0) i (delete all)
D(0, j) j (insert all)

Fill the grid row by row; the answer is D(n,m). O(n·m). Trace the chosen arrows back to reconstruct the actual alignment.

Knapsack (0/1 and unbounded)

Items with weights and values, a sack of capacity W, maximize value. The greedy value-per-weight trick doesn't work for whole items — this needs DP.

Variant Recurrence
With repetitions V(w) = max over items i of [ V(w - wi) + vi ]
Without (0/1) V(w, i) = max( V(w, i-1),   V(w - wi, i-1) + vi )

The 0/1 version adds a second axis (which items are allowed) so each item is used at most once. Both run in O(n · W) — the "cut-and-paste" argument proves an optimal sack minus one item is still optimal for the smaller capacity.

Placing parentheses — optimal order

Maximize an arithmetic expression like 1 + 2 - 3 × 4 - 5 by choosing the order of operations. Brute force is n! orders (20 digits > a year). DP instead solves every sub-expression once, tracking both the max and min of each (a minus times a min can become the max), and combines them — O(n³).

The pattern every time: define the subproblem, write the recurrence, fill the table in dependency order, then read back the solution.
