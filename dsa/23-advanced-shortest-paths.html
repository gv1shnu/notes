Advanced Shortest Paths

For a single source→target query on a huge graph (non-negative weights), plain Dijkstra is too slow — it fans out in a growing circle around s, scanning millions of irrelevant vertices. The first easy win is early stopping: quit the moment t is extracted. These techniques shrink the search far more.

Bidirectional Dijkstra

Run two searches at once — forward from s, backward from t (over reversed edges) — alternating turns, and stop when they meet in the middle. Two radius-r circles cover roughly half the area of one radius-2r circle, so on road maps it's about a 2× speedup. On small-world graphs (social networks) the one-big-circle is enormous but the two meeting circles are tiny — thousands× faster.

A* — Dijkstra with potentials

Add a potential π(v) estimating the remaining distance v→t (e.g. straight-line / GPS distance). Reweight every edge: 

ℓπ(u, v) = ℓ(u, v) − π(u) + π(v)

The π terms telescope along any s→t path, so they shift every path's length by the same constant — shortest paths are unchanged. Running Dijkstra on these weights picks the vertex minimizing dist[v] + π(v): the one that looks closest to the goal, so the search aims at t instead of spreading evenly. The catch — π must be feasible (all ℓπ ≥ 0, i.e. π a lower bound on the true remaining distance), or Dijkstra's non-negativity assumption breaks.

In code, A* is just Dijkstra with the priority changed from g = dist[v] to f = dist[v] + h(v): 

import heapq

def a_star(graph, s, t, h):         # h(v) = lower-bound estimate to t
    dist = {s: 0}
    pq = [(h(s), s)]                # priority f = g + h
    while pq:
        _, u = heapq.heappop(pq)
        if u == t: return dist[u]   # target settled -> done
        for v, w in graph[u]:
            nd = dist[u] + w
            if nd < dist.get(v, float('inf')):
                dist[v] = nd
                heapq.heappush(pq, (nd + h(v), v))
    return float('inf')             # unreachable

vector<vector<pair<int,int>>> graph;   // graph[u] = {(v, w), ...}

long long a_star(int n, int s, int t, function<long long(int)> h) {
    const long long INF = 1e18;
    vector<long long> dist(n, INF);
    dist[s] = 0;
    priority_queue<pair<long long,int>,
        vector<pair<long long,int>>, greater<>> pq;
    pq.push({h(s), s});             // priority f = g + h
    while (!pq.empty()) {
        auto [f, u] = pq.top(); pq.pop();
        if (u == t) return dist[t]; // target settled -> done
        for (auto [v, w] : graph[u]) {
            long long nd = dist[u] + w;
            if (nd < dist[v]) {
                dist[v] = nd;
                pq.push({nd + h(v), v});
            }
        }
    }
    return INF;                     // unreachable
}

Bidirectional Dijkstra is two of these searches sharing a best-meeting distance; contraction hierarchies add an offline preprocessing pass — both are long enough that in practice you lean on a library (see notes).

Contraction hierarchies

The big gun for road networks. Long trips climb a hierarchy of roads — street → highway → interstate → highway → street — so node importance rises then falls along a shortest path. Split the work into two phases:

Phase What happens Cost
Preprocess (once) order all nodes by importance, contract them adding shortcut edges that preserve distances hours/days — offline
Query bidirectional search that only ever moves to more important nodes milliseconds

Because both halves only climb upward in importance and meet at the single most-important node on the route, the explored graph is tiny. Same answer as Dijkstra, thousands× (and more) faster — this is the preprocess-then-serve pattern real map services (Google, Yandex) ship to production.

Method Idea Road-network speedup vs Dijkstra
Early stopping quit when t is settled modest
Bidirectional meet in the middle ~2× (huge on social graphs)
A* aim at the target with π depends on heuristic quality
Contraction hierarchies preprocess + climb importance thousands×+

Language notes

  • A* is Dijkstra with priority dist + h — reuse heapq. A consistent (monotone) heuristic keeps this simple version correct.
  • Don't hand-roll the hard ones: networkx ships astar_path and bidirectional_dijkstra for small/medium graphs.
  • Real road-scale routing (contraction hierarchies, transit-node routing) → a C-backed library (e.g. routingkit bindings, OSRM), not pure Python.
  • std::function for the heuristic is convenient; template it for speed in hot loops.
  • Bidirectional Dijkstra = two priority_queues plus a shared best-meeting distance and a proper stopping rule.
  • Production routing uses dedicated engines (RoutingKit, OSRM) that build the contraction hierarchy offline — you query, not rebuild.
