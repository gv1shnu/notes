Hash Tables

Store a map (keys → values) with near-O(1) HasKey / Get / Set. A hash function h turns a key into an index in 0..m-1; you store the pair at that index. The snag: two keys can hash to the same index — a collision.

Chaining

Make each table cell a (doubly-)linked list. On a collision, just append to that cell's list ("chain"). Lookup = hash to the cell, then scan its chain. 

class HashTable:
    def __init__(self, m=8):
        self.m = m
        self.buckets = [[] for _ in range(m)]   # chains

    def _h(self, key):
        return hash(key) % self.m

    def set(self, key, value):                  # O(1) average
        chain = self.buckets[self._h(key)]
        for i, (k, _) in enumerate(chain):
            if k == key: chain[i] = (key, value); return
        chain.append((key, value))

    def get(self, key):                         # O(1) average
        for k, v in self.buckets[self._h(key)]:
            if k == key: return v
        return None

struct HashTable {
    int m;
    vector<list<pair<int,int>>> buckets;       // chains
    HashTable(int m = 8) : m(m), buckets(m) {}

    int h(int key) { return ((key % m) + m) % m; }

    void set(int key, int value) {             // O(1) average
        auto& chain = buckets[h(key)];
        for (auto& kv : chain) if (kv.first == key) { kv.second = value; return; }
        chain.push_back({key, value});
    }
    bool get(int key, int& out) {           // O(1) average
        for (auto& kv : buckets[h(key)]) if (kv.first == key) { out = kv.second; return true; }
        return false;
    }
};
Quantity Cost
HasKey / Get / Set O(1 + c), c = longest chain
Memory O(n + m)

Watch it run

Insert keys into m = 7 buckets, h(key) = key % 7. Most land in an empty bucket; when two keys share an index it's a collision — they just chain downward. Watch the chains grow as the load factor climbs. Reset picks new keys.

Load factor

α = n / m (items ÷ cells). Too small → wasted space; too big → long chains, slow. Keep it in the sweet spot, roughly 0.5 ≤ α ≤ 1.

Universal hashing

No single hash function is good for all inputs (an adversary can pick keys that all collide). The fix is randomization, like quicksort's random pivot: define a universal family of hash functions — one where any two distinct keys collide with probability ≤ 1/m — and pick one at random.

With a random universal hash Expected
longest chain O(1 + α)
Get / Set / HasKey O(1) on average (α kept constant)

Rehashing

Don't know n in advance? Borrow the dynamic-array trick: when α crosses a threshold (say 0.9), double the table, pick a fresh hash function, and re-insert every key. A single rehash is O(n), but it's rare — so amortized cost per operation stays O(1).

Language notes

  • dict and set use open addressing (probing, not chaining) with a compact, insertion-ordered layout; O(1) average get/set, resize near 2/3 full.
  • Keys must be hashable (immutable) — __hash__ + __eq__. String hashing is randomized per process (PYTHONHASHSEED) to thwart collision DoS.
  • collections.defaultdict / Counter are dict subclasses for the common "missing key" and tallying patterns.
  • std::unordered_map / unordered_set use bucket chaining; O(1) average, O(n) worst. Tune with reserve/max_load_factor.
  • std::map / set are a different structure — balanced (red-black) trees, O(log n), kept sorted. Pick them when you need ordering or range queries.
  • Custom key types need a hash functor (std::hash specialization) plus operator==.
