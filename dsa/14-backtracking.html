Backtracking

Difficulty: Hard

Build a solution one choice at a time, and the moment a partial choice can't lead to a valid answer, undo it and try the next option. It's a depth-first walk over the tree of all choices — with the power to prune whole branches early. The natural fit for combinatorial search: permutations, subsets, N-Queens, Sudoku, mazes.

The recipe: choose, explore, un-choose

Step What
1. Choose make one move, extend the partial solution
2. Explore recurse on the rest
3. Un-choose undo the move (the "backtrack") so the next option starts clean

That third step — restoring state on the way back up — is what makes it backtracking rather than plain recursion. Here it generates every subset: 

def subsets(nums):
    res, path = [], []
    def backtrack(start):
        res.append(path[:])              # record current subset
        for i in range(start, len(nums)):
            path.append(nums[i])         # choose
            backtrack(i + 1)             # explore
            path.pop()                   # un-choose
    backtrack(0)
    return res

void backtrack(int start, vector<int>& nums,
               vector<int>& path, vector<vector<int>>& res) {
    res.push_back(path);                 // record current subset
    for (int i = start; i < (int)nums.size(); ++i) {
        path.push_back(nums[i]);         // choose
        backtrack(i + 1, nums, path, res);  // explore
        path.pop_back();                 // un-choose
    }
}

Pruning is the whole point

The decision tree is exponential — trying every branch is brute force. Backtracking earns its keep by killing branches early: add a check before "explore" and skip choices that already violate a constraint (a queen on a shared diagonal, a digit breaking a Sudoku row). The earlier you prune, the more of the tree you never walk.

Cost
Brute force (no pruning) explore every leaf
Backtracking (with pruning) exponential worst case, but often vastly less in practice

It's recursion with an undo step, so it inherits the call stack — and runs on almost anything, which is why the roadmap lists it under "any."

Watch it run

N-Queens — place one queen per row so none attack. Step through: a queen goes down a row (green ♛); a column under attack is skipped (red); when a row has no safe column the search backtracks and removes the previous queen (orange) to try its next option. That undo is backtracking. Reset replays.

Language notes

  • For the classic enumerations, itertools already has it — permutations, combinations, product — faster than a hand-rolled loop.
  • path[:] (or list(path)) makes a copy when recording; appending path itself stores a reference that later pops will mutate.
  • std::next_permutation walks permutations in-place without recursion.
  • Pass path/res by reference to avoid copying the whole vector at every call; push_back(path) copies only when you record a hit.
