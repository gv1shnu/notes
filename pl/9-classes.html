Classes & Objects

Difficulty: Medium

Once data and the functions that act on it belong together, bundle them. A class is a blueprint; an object is one thing built from it. The class Player defines what every player has and does; each actual player is an instance. 

class Player:
    def __init__(self, name, score):   # constructor — runs on creation
        self.name = name                # attribute (data)
        self.score = score

    def promote(self, points):         # method (behaviour)
        self.score += points

p = Player("TenZ", 842)   # create an instance
p.name                       # 'TenZ'
p.promote(50)                # p.score is now 892

The two names that confuse everyone

Name What it is
__init__ the constructor — called automatically when you write Player(...), sets up the object's starting attributes
self the instance itself — the first parameter of every method, how an object refers to its own data

You never pass self explicitly — p.promote(50) quietly becomes Player.promote(p, 50), and p arrives as self. Inside a method, self.score reads/writes this object's score, keeping each instance's data its own.

Why bother

You could keep a player as a dict and pass it to loose functions — but a class keeps the data and the operations on it in one place, so the rest of your code says p.promote(50) instead of juggling dictionaries and helper functions. That bundling is the whole point of objects, and it pays off most when classes start building on each other.
