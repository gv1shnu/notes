Error Handling

Difficulty: Medium

We've built values, control flow, and objects assuming everything goes right. It doesn't. Things go wrong — a user types letters where you expected a number, a file isn't there, you divide by zero. Python signals these with exceptions: by default they crash the program. Handling them means catching the crash and deciding what to do instead.

try / except

try:
    age = int(input("Age? "))   # might fail if not a number
except ValueError:
    print("That wasn't a number.")   # runs only on that error

Code that might fail goes in try; the recovery goes in except. Catch the specific exception you expect — a bare except: swallows everything (including typos and Ctrl-C), hiding bugs you'd want to see.

The full shape

Block Runs when
try always — the risky code
except an error of that type occurred
else no error occurred
finally always — error or not (cleanup) 
try:
    f = open("data.txt")
except FileNotFoundError:
    print("No file.")
else:
    print(f.read())     # only if open() succeeded
finally:
    print("done")        # cleanup, no matter what

Raising your own

def withdraw(balance, amount):
    if amount > balance:
        raise ValueError("insufficient funds")
    return balance - amount

Use raise to signal a problem your own code detects — better to fail loudly at the source than return a wrong answer. The caller can then try/except it. (Common built-ins you'll meet: ValueError, TypeError, KeyError, IndexError, ZeroDivisionError.)
