Modules & Libraries

No one writes everything from scratch. A module is just a .py file full of code you can pull into another with import — whether it's Python's own batteries, someone else's package, or your own file.

Importing

import math                 # whole module
math.sqrt(16)               # 4.0   — reach in with the name

from random import randint   # just one name
randint(1, 6)               # a die roll, no 'random.' prefix

import numpy as np           # aliased (common for big libs)
Form Use
import math keeps the namespace — math.sqrt, clear where it came from
from math import sqrt pulls one name in directly — shorter, but can clash
import numpy as np a short alias for a long name

The standard library alone covers a lot — math, random, datetime, json, os. Beyond that, pip install <name> grabs third-party packages (numpy, requests, …) from the Python Package Index.

Your own module

Any file you write is importable by its filename. Put helpers in tools.py, then elsewhere: 

from tools import greet   # uses greet() defined in tools.py

Imported vs run directly

def main():
    print("running!")

if __name__ == "__main__":   # only when run directly, not when imported
    main()

When you run a file, Python sets its __name__ to "__main__"; when you import it, __name__ is the module's name instead. That one check lets a file double as both a runnable script and an importable library — its functions load on import, but the demo code only fires when you run it yourself.

That's the core of Python: values in containers, decisions and loops, organized with functions and objects, kept safe with error handling, and scaled with the world's modules. The rest is practice.
